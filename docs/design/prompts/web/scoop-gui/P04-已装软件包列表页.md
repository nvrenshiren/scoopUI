## 1. 设计目标与基调

P04 是用户进入主界面后停留时间最长的页面(`nav.installed` 为侧边栏默认选中项),核心任务是让用户在**一屏之内**完成三件事:确认"我装了什么"、发现"哪些该更新了"、决定"要不要现在处理"。据此本页设计基调取设计系统 §1 中以下三条原则:

- **即时反馈**:过期项必须"一眼可见",不需要用户逐行核对版本号——过期行整体降调染色(`bg-warning/5`),而不是仅在某个小徽标上做提示。
- **静默高级**:六列表格信息密度已经足够高,不额外引入卡片式列表或缩略图;克制用色——只有"过期"与"忙碌中"两种语义状态会打破默认的中性配色。
- **终端原生感**:名称、版本号、可用版本一律 `font-mono`——这些是"类命令行输出"的数据,与设计系统 §4 字体角色表一致,也呼应用户从 `scoop list`/`scoop status` 命令行迁移过来的视觉记忆。

本页**不**是一个纯只读列表——它同时承载 F06(卸载)、F07(单包更新)、F08(批量更新)三个写操作入口,因此设计上必须让"只读浏览"与"发起写操作"两类动作在视觉权重上明确分层:主 CTA(更新全部过期)与次要巡检动作(检查更新)放在页头,行内单个动作(更新/卸载)收敛到表格操作列,避免整页出现两套并列的"主按钮"。

## 2. 页面骨架总览

对应设计系统 §10.2"列表页(P04/P05/P06)"骨架,P04 的具体化落地为四段纵向堆叠区域:

```
┌───────────────────────────────────────────────────────────────┐
│ .bg-grid + .bg-glow-top(仅覆盖本区域,不延伸到下方表格)         │
│  ┌───────────────────────────────┐  ┌────────────┐┌──────────┐│
│  │ H1 已装软件包                    │  │ 检查更新    ││更新全部过期││
│  │ 说明文案(过期规则一句话)          │  │(outline)  ││(default)  ││
│  └───────────────────────────────┘  └────────────┘└──────────┘│
├───────────────────────────────────────────────────────────────┤
│ [搜索输入框(左侧图标,flex-1)]                [刷新 icon 按钮] │  ← SubToolbar
├───────────────────────────────────────────────────────────────┤
│ TableWrap(圆角卡片边框,仅纵向滚动)                             │
│  thead(sticky,bg-card)  名称|版本|可用版本|来源桶|状态|操作     │
│  tbody 行(h-10,过期行 bg-warning/5)                           │
└───────────────────────────────────────────────────────────────┘
```

与设计系统 §10.2 原文的对应关系需要特别说明(源码实际拆分方式,详见第 11 节"已知偏离记录"第 1 条):**页头(PageHeader)右侧承载的是"检查更新"与"更新全部过期"两个主要动作按钮,而不是搜索/刷新**;搜索输入框与图标刷新按钮下沉到独立的 SubToolbar 行。这样拆分的产品意图是:页头是"全局巡检"动作(检查是否有更新、批量处理更新),SubToolbar 是"在当前已装列表内定位"的辅助动作(过滤、手动刷新列表本身)——二者语义不同,不应该挤在同一行。

Footer(设计系统骨架中"分页预留"区块)本页**不渲染**——已装软件包数量通常在几十到一两百之间,一屏纵向滚动即可覆盖,分页不在本模块 MVP 范围内(详见第 11 节第 4 条)。

## 3. PageHeader 区

- 容器:`px-7 pt-6 pb-4`,叠加 `.bg-grid` + `.bg-glow-top`(相对定位,网格在下、内容在上)。
- 左侧(占满剩余宽度):
  - `<h1>` 标题,`text-2xl font-semibold`(继承 `font-heading`):「已装软件包」/「Installed Apps」。
  - 标题下方一行说明文案,`text-[13px] text-muted-foreground`,内容是"过期"标记规则的一句话解释,让首次使用者无需查文档就懂徽标含义。
- 右侧两个按钮,顺序从左到右:
  1. **检查更新**(`variant="outline"`,default size):图标默认是 `RefreshCw`,触发中(`loading.status`)替换为 `LoaderCircle` 旋转 + `text-primary`,此时按钮本身仍可点击但重复点击应是幂等的(不产生第二次并发请求语义上的问题,由 developer 层保证)。这是本页**唯一**对应 F03(过期检查)的显式触发入口——对应 `scoop status`。
  2. **更新全部过期**(default variant,`CircleArrowUp` 图标):当前无过期项时 `disabled`;有过期项时按钮尾部附加一枚深色底浅色字的计数徽标(`bg-black/25 text-xs rounded-sm px-1.5`,不是标准 Badge 组件,是按钮内嵌的计数胶囊),显示当前过期项数量。点击后**不**立即执行,而是打开二次确认对话框(见第 7 节)。

两个按钮之间无需分割线,靠 `gap-2` 的间距与 outline/default 两种视觉权重的差异自然区分主次动作。

## 4. SubToolbar 区

- 容器:`px-7`,与 PageHeader 共享左右内边距,纵向 `gap-3` 与上方分隔。
- 左侧:搜索输入框(`flex-1`,内部左侧叠加 `Search` 图标,`pl-9` 让文字避开图标),placeholder 为"过滤已装软件包…"/"Filter installed apps…"。这是**纯前端本地过滤**(按名称子串、大小写不敏感),不触发任何 IPC 调用,因此不需要 loading 态、不需要防抖提示——用户每次按键都应看到即时的表格行数变化。
- 右侧:一个 `icon` size 的 `outline` 按钮(仅 `RefreshCw` 图标,`aria-label="刷新"`),点击后同时触发"已装列表"与"过期检查"两条数据源的刷新(即 F01 已装部分 + F03 一起刷新,而不仅仅是 F03)——这是本页除"检查更新"外**第二个**触发数据刷新的入口,二者的差别是:"检查更新"只刷新过期比对,这个 icon 按钮是"整页手动重新拉取"的兜底动作(常用于怀疑数据不同步、或从写操作返回后想确认全貌的场景)。

## 5. 表格区

沿用设计系统 §8.6 表格形态与 `TableWrap` 容器规格:

| 规格项 | 取值 |
|---|---|
| 容器 | 圆角 `radius-lg`,`1px solid border`,`bg-card`;仅纵向滚动(`overflow-y-auto overflow-x-hidden`),**不出现横向滚动条**——这是项目级硬约束(见 `CLAUDE.md`),超宽内容一律 ellipsis 截断而不是横向滚动 |
| 表头 | `sticky top-0`,`bg-card`,行高 `h-10`,底部 `1px border` |
| 行高 | `h-10`(40px),仅底部分隔线,无纵向分隔线 |
| 列宽策略 | 每个单元格 `max-w-[360px]` + ellipsis;六列中"名称""版本""可用版本"三列使用 `font-mono`,与设计系统 §4"数字/代码用等宽字体防 layout shift"的原则一致 |

列定义(从左到右,对应 `InstalledApp` / `StatusEntry` 数据契约,见 `docs/architecture/api/scoop-gui.md` §3.5/§3.6):

| # | 列标题(zh/en) | 数据来源 | 呈现规则 |
|---|---|---|---|
| 1 | 名称 / Name | `InstalledApp.name` | `font-mono font-medium`,整行的点击热区触发进入 P07(见第 9 节) |
| 2 | 版本 / Version | `InstalledApp.version` | `font-mono text-muted-foreground`,本机当前已装版本 |
| 3 | 可用版本 / Latest | `StatusEntry.latestVersion` | 仅当该包过期时显示,`font-mono text-warning`;非过期时显示纯文本"—"(`text-subtle`),不留空白 |

| 5 | 状态 / Status | 综合 `StatusEntry` 比对结果与忙碌任务集合 | 见第 6 节"行状态矩阵",三态 Badge |
| 6 | 操作 / Actions | — | 右对齐,按钮组见第 7 节;整列 `onClick` 需要阻止冒泡,避免误触发整行的"进入详情"点击 |

## 6. 行状态矩阵(核心规格)

一行的视觉表现由两个**相互独立**的判定共同决定:「是否过期」(`StatusEntry` 比对结果)与「是否忙碌」(该包名当前是否存在排队中/执行中的 InstallJob)。二者分别控制不同的视觉层,不要互相覆盖:

| 情形 | 行背景 | 状态列 Badge | 操作列 |
|---|---|---|---|
| 最新 · 空闲 | 默认(无染色) | `success` variant,`Check` 图标,「最新」 | 仅「卸载」按钮(ghost variant,`text-destructive`) |
| 过期 · 空闲 | `bg-warning/5`(整行浅染) | `warning` variant,`TriangleAlert` 图标,「过期」 | 「更新」(sm size,default variant)+「卸载」(ghost) |
| 任意判定 · 忙碌中(排队中/执行中) | **沿用该行原有的过期/最新底色,不额外新增或清除** | `info` variant,旋转 `LoaderCircle` + 当前任务种类文案(如「更新」「卸载」) | 该行全部操作按钮 `disabled`(仍渲染,视觉降为 `opacity-50`,不是隐藏) |

这条矩阵里最容易被忽略、也是本次核对中重点确认过源码行为的一点是:**"忙碌"只改变状态列 Badge 的显示优先级,不影响行背景色的判定**。也就是说,一个"过期中"的软件包如果正在被更新,它的行背景依旧保留过期的浅染警示色,状态列同时切换成"执行中"的旋转徽标——两个视觉信号叠加呈现,而不是互斥;这与设计系统 §9"按钮禁用态"硬约束("长时操作中除取消外所有按钮 disabled")共同构成本页忙碌态的完整表现。原型第 8 行(neovim)专门用于演示这一叠加状态。

## 7. 操作与二次确认策略

本页三个写操作入口的确认策略**并不统一**,这是源码现状(详见第 11 节第 5 条),设计提示词按现状登记:

| 操作 | 触发位置 | 二次确认 | 确认弹窗内容 |
|---|---|---|---|
| 单包更新(F07) | 表格行内「更新」按钮 | **无**——点击后直接入队(`enqueue("update", name)`),立即进入忙碌态 | 不适用 |
| 单包卸载(F06) | 表格行内「卸载」按钮 | 有——`ConfirmDialog`,`destructive` 语气 | 标题「卸载确认」;正文「确定要从本机卸载 {name} 吗?」;底部 `secondary` 取消 + `destructive` 卸载 |
| 批量更新全部过期(F08) | 页头「更新全部过期」按钮 | 有——`ConfirmDialog`,常规语气(非 destructive) | 标题「批量更新确认」;正文「将按顺序更新以下 {n} 个过期软件包,每个软件包对应一条任务:」;下方追加一个等宽字体的清单框(`font-mono text-xs`,逐行列出本次将更新的包名);底部 `secondary` 取消 + `default`「更新全部过期」确认 |

确认弹窗均遵循设计系统 §8.4 Dialog 规格(居中、`bg-card`、`radius-xl`、`shadow-xl`、`bg-black/60 backdrop-blur-sm` 遮罩),支持 `Enter` 确认、点击遮罩或 `Esc` 取消。批量更新确认后,前端按 flow §1.6 的业务语义**逐包生成独立的 InstallJob**(而不是单条"批量任务"),因此确认弹窗关闭后,用户会看到任务面板(P08)里出现与过期项数量相同的多条任务,这一点在设计提示词层面需要提前设定预期,避免被误解为"应该只有一条汇总任务"。

卸载确认弹窗有一个跨页面复用细节:P07(软件包详情对话框)里点击「卸载」不会在 P07 内部弹出确认,而是关闭详情对话框、把视图切回本页、再打开本页这同一个 `ConfirmDialog` 实例——即卸载确认弹窗**只存在一份**,由 P04 承载,P07 是它的另一个触发来源而非另一份独立弹窗。

## 8. 空状态与错误态(P10 子状态)

本页在两种数据缺失场景下分别呈现 `EmptyState` 组件(表格区域被整体替换,页头与 SubToolbar 保留常驻):

| 场景 | 图标 | 标题 | 说明 | 操作 |
|---|---|---|---|---|
| 已装列表为空(首次使用) | `Package`(中性色) | 「本机还没有通过 Scoop 安装任何软件包」 | 「前往"浏览软件包"查找并安装第一个软件包」 | 无按钮(依赖侧边栏导航自行前往 P05) |
| 已装列表读取失败 | `TriangleAlert`(warning 色调容器) | 「无法读取数据」 | 「本机 Scoop 当前不可用……请检查 scoop 命令是否正常后刷新重试」 | `secondary` 「刷新」按钮,直接重试 F01+F03 两条刷新 |

过滤输入导致的"零匹配"(用户主动搜索出的空结果,而非真实数据为空)**复用**上述同一个 `EmptyState`——源码里 `rows.length === 0`(`rows` 已经是过滤后的数组)与"已装列表真的为空"共用同一段判断和同一套 `installed.empty`/`installed.emptyHint` 文案,并不区分"本机真的一个包都没装"与"过滤条件太窄导致零匹配"这两种不同的业务含义。也就是说,用户在有 8 个已装软件包的机器上,只要把过滤框打成一个谁都不匹配的关键字,看到的提示会是「本机还没有通过 Scoop 安装任何软件包 / 前往"浏览软件包"查找并安装第一个软件包」——这段文案在"零匹配"场景下其实是不准确的。本设计提示词与随附原型如实复现这一源码现状(不擅自加做"零匹配应有独立提示"这类改进),该点已在第 11 节"已知发现"中登记,供后续统一裁决是否要在源码里补一条独立的"过滤零匹配"文案。

过期比对(F03)单独失败时**不**整页转入错误态——已装列表本身若能正常读取,列表照常展示,只在 Sonner Toast 中提示「过期检查失败」,不阻断用户查看已装清单本身。这一行为对应模块 PRD §4 决策口径"F19 涵盖异常处理但不引入新业务实体",过期比对是已装列表的增强信息而非前置条件。

## 9. 与相邻页面的流转出口

- 点击表格行(非操作列区域)→ 进入 **P07 软件包详情对话框**,以居中 Dialog 形式覆盖当前页面,本页本身不需要维持"选中行"的持久高亮(设计系统 §8.6 提到的"选中行 `bg-primary/10` + 左侧 2px 边线"通用规则,在 P04 这种"点击即弹出模态详情"的交互下没有对应的持久态可呈现,详见第 11 节第 2 条附带说明)。
- 点击「更新」「卸载」或确认「更新全部过期」→ 对应 InstallJob 被创建,**P08 任务面板**自动展开(非模态浮层,不打断本页浏览),本页继续可见可交互。
- 所有写操作成功后,只刷新被操作的单个软件包(`scoop list <name>` 子串查询后精确匹配),不触发本页整体重新拉取——这是项目级硬约束(`CLAUDE.md`"单包任务成功后只做单包刷新"),设计上因此不需要为"整行替换/整表重排"设计过渡动效,只需要保证单行数据更新时有一个短暂的高亮或渐变提示(具体时长由 developer 决定,设计提示词不做强制)。

## 11. 已知偏离记录(如实登记,不修改上游契约)

以下条目是核对设计系统 v2 文字描述、页面 PRD 与源码(`src/pages/InstalledView.tsx`、`src/i18n.ts`、`src/components/ui/badge.tsx`、`src/index.css`)三方一致性时发现的出入。按本任务要求,均不反向修改设计系统/页面 PRD/源码,只在本文档如实记录,供后续统一裁决:

1. **PageHeader/SubToolbar 的按钮分配**:设计系统 §10.2 骨架文字写"PageHeader: 标题+描述+右侧工具栏(搜索/刷新)",但源码把"检查更新""更新全部过期"两个主操作按钮放在页头右侧,搜索输入框与图标刷新按钮下沉到独立的 SubToolbar 行。本文档与随附原型按源码实现记录(见第 2、3、4 节)。
2. **过期标记图标的位置**:设计系统 §8.6 表格小节写"过期行:`bg-warning/5` + 行首 `AlertTriangle` 图标",源码实际把 `TriangleAlert` 放在"状态"列的 Badge 内部(遵循 §8.9 Badge 规格的 warning variant 自带图标),而不是在行首单独放置图标;行本身除背景染色外没有独立图标元素。本文档第 6 节按源码记录。
3. **Badge variant 数量**:设计系统 §8.9 列出 `default`/`success`/`warning`/`destructive`/`outline` 共 5 个 variant,源码 `badge.tsx` 实际存在第 6 个 variant `info`(`border-info/30 bg-info/10 text-info`),P04(以及 P07、P08)的"忙碌/执行中"状态徽标均依赖这个未在设计系统中登记的 variant。
4. **Footer(分页预留)未实现**:设计系统 §10.2 骨架列出"Footer: 分页(预留)"区块,源码 `InstalledView.tsx` 当前完全不渲染任何 Footer/分页/计数区域(表格是该列最后一个元素)。分页也不在模块 PRD 决策记录范围内,推测属于后续阶段能力;本文档与原型同样不呈现 Footer。
5. **单包更新是否需要二次确认(本次核对中最值得关注的一处)**:页面 PRD 第 4 节场景 5"单包更新"的触发描述为"用户对过期软件包选择更新并确认",与场景 4(卸载)、场景 6(批量更新)采用相同的"选择并确认"措辞、隐含三者都应有二次确认步骤。但源码中卸载与批量更新都渲染了 `ConfirmDialog`,唯独单包更新按钮 `onClick={() => void enqueue("update", app.name)}` 直接触发任务入队,没有任何确认弹窗或二次交互。本文档与原型按源码"单包更新 = 一键直达,无二次确认"如实呈现(见第 7 节),该出入已在本次任务的最终交付说明中一并报告给上游,不在本文档或页面 PRD 中擅自改写。
6. **"更新全部过期"禁用态缺少说明文案**:`src/i18n.ts` 中存在 `installed.noOutdated`("当前没有过期的软件包"/"No outdated apps right now")这一条目,但源码全文未引用该 key——按钮仅以 HTML `disabled` 属性隐式表达不可点击,没有 tooltip 或提示文案说明原因,该 key 目前是未被消费的"死文案"。本文档按源码现状记录,不在原型中新增 tooltip 交互(那将是对现有行为的扩展,超出"如实登记"的任务边界)。
7. **"已装列表真空"与"过滤后零匹配"共用同一段空状态文案**:源码 `rows.length === 0` 判断(`rows` 为过滤后的数组)统一走 `<EmptyState title={t("installed.empty")} hint={t("installed.emptyHint")} />`,未对"本机确实一个包都没装"与"用户把过滤框打成零匹配关键字"这两种业务含义不同的场景做区分。后者场景下继续呈现"本机还没有通过 Scoop 安装任何软件包"文案存在误导性(用户机器上明明有软件包,只是被过滤条件排除了)。本文档与原型第一版曾经把"零匹配应有独立提示"当作新设计写入,经与源码逐行核对后判定这不是源码现状,已改为如实复现源码的复用行为(见第 8 节),并在此登记为待裁决项。
