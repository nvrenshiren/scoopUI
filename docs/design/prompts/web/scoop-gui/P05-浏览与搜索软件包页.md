## 1. 设计目标与基调

P05 承载 F01(可装部分)/ F02 / F04 / F05 / F17 / F19,是用户"发现新软件包"的唯一入口。与 P04(已装列表,面向"我已经有什么")不同,P05 面向"还能装什么",设计基调需要传达:

- **浏览优先,搜索是加速器**:进入页面默认就是完整可装列表(设计系统 §10.2 列表页骨架),搜索框不是必经步骤,而是列表很长时的过滤工具——这决定了搜索框在视觉上应该"轻量常驻",而不是需要用户主动唤起的独立浮层(取舍详见第 4 节)。
- **克制的状态提示**:本页不引入新的图形语言,复用 P04 已经建立的表格 + Badge 语言(设计系统 §8.6 / §8.9),让用户在两个列表页之间零学习成本切换。
- **安装是轻量操作**:不同于卸载(destructive,需要二次确认),本页的"安装"是纯增量操作——按钮点击后立即入队,进度由任务面板(P08)承接,不打断浏览动作(设计取舍见第 6 节)。

## 2. 整体骨架与网格背景

页面严格遵循设计系统 §10.2"列表页(P04/P05/P06)"骨架,自上而下四段:

```
[ PageHeader:标题 + 描述 + 右侧刷新按钮,叠加 .bg-grid + .bg-glow-top ]
[ SubToolbar:搜索框(全宽,含左侧搜索图标)]
[ Table:表头 sticky + 行 h-10 + 状态列 Badge ]
[ Footer:"显示 X / Y 条" 统计文案(容量提示,无分页控件) ]
```

- 网格背景(设计系统 §9"网格背景(科技感)"硬约束)只叠加在 **PageHeader 区块**,不铺满全页——与 P04 已装列表页保持同一处理方式,不在表格区域叠加网格纹理,避免干扰表格可读性。
- 页面内容本身不使用 Card 包裹(不同于 P07 对话框);表格通过 `TableWrap`(§8.6 表格容器)自带边框与圆角,充当本页唯一的容器视觉单元。
- 页面主区**不**重复侧边栏顶部的品牌渐变线(§9"顶部品牌线"只属于 Sidebar,不在每个子页面重复)。

## 3. PageHeader 设计

| 元素 | 规格 |
|---|---|
| 标题 | `<h1>`,`font-heading`(Space Grotesk)、`text-2xl`(1.5rem)、字重 600 |
| 描述 | 标题下方 4px 间距,`text-[13px]`、`--fg-muted` |
| 刷新按钮 | 右对齐,`outline` variant(§8.1);常态图标为 RefreshCw,`loading=true` 时替换为 `LoaderCircle` 旋转图标,按钮本身 `disabled`(§9"按钮禁用"硬约束) |
| 背景叠加 | `.bg-grid` 铺满该区块 + `.bg-glow-top` 径向渐变叠加在区块内(不外溢到表格区) |

文案(i18n key,zh 默认值见第 12 节):`browse.title` = "浏览软件包"、`browse.desc` = "已添加桶中的可安装软件包;数据来源于本机 Scoop,不联网发现新软件源。"——描述文案本身承担一次隐含的合规提醒(呼应 `project.md` 软件源唯一性硬约束),设计上不需要额外加图标或强调色,与 P04 描述文案保持同等的克制视觉权重。

## 4. 搜索框设计(与 §8.11 的取舍说明)

**本页搜索形态 = §8.3 Input(带前缀图标的常驻输入框),不是 §8.11 Command/cmdk。** 设计系统 §8.11 描述的是一个由 `Ctrl+K` 或顶部 SearchBar 触发的**居中浮层**搜索(结果列表悬浮于内容之上,`Esc` 关闭)。P05 的搜索业务性质与之不同,不适用该形态:

- 数据在进入页面时已经**全量**加载到本地(`scoop search` 以无查询参数一次性拉取全部可装软件包,`refreshAvailable()`),搜索行为是对已加载数据的**纯前端即时过滤**,不产生新的命令调用——不存在"等待搜索结果"的异步态,不需要浮层承载"加载中"的结果列表。
- 过滤结果需要与"浏览"共用同一张表格(用户缩小范围后仍要能直接点击行进入详情或点安装),浮层弹窗反而会在"过滤"与"操作结果行"之间多出一次开关。

因此本页搜索框规格为:

| 项 | 规格 |
|---|---|
| 容器位置 | PageHeader 下方,独占一行,宽度 100% |
| 组件 | §8.3 Input,高度 40px,`bg-secondary`(即 `--bg-overlay`)背景 + `--border` 边框 |
| 前缀图标 | `Search`,`size-4`,`--fg-subtle` 色,绝对定位于输入框左侧(左内边距让位) |
| 占位符 | `browse.searchPlaceholder` = "按名称搜索软件包…" |
| 聚焦态 | 边框变 `--primary` + `ring-primary` 外环微光(§8.3 通用规则) |
| 过滤时机 | 每次 `input` 事件立即过滤(无防抖 debounce,无 loading 态)——纯本地字符串匹配,响应必须瞬时,防抖反而造成"察觉到的卡顿" |
| 匹配规则 | 大小写不敏感的**子串匹配**,只匹配"名称"字段(不匹配版本号/来源桶) |
| 清空 | 用户手动删除输入内容即可恢复完整列表;不额外提供"×"清空按钮,与 P04 过滤框行为一致 |

搜索框**在任何数据态下都保持渲染**(加载中 / 空结果 / 读取失败态下依然可见可输入,只有其下方内容区在四态之间切换)——这是本页与"整页替换式"页面(如 P01)的关键差异:P05 的空/错态只替换"结果区",搜索这一操作入口必须持续可达。

## 5. 结果表格设计

沿用 §8.6 表格规格(行高 `h-10`、表头 `sticky`、仅水平分隔线),列定义如下(与 `SearchResult` 数据结构对应):

| 列 | 数据字段 | 对齐 | 排版 |
|---|---|---|---|
| 名称 | `name` | 左 | `font-mono font-medium`——包名本质是命令行标识符,与 P04 已装列表名称列排版一致 |
| 版本 | `version` | 左 | `font-mono text-muted-foreground`(§4"数字/版本号用等宽字体防 layout shift"原则) |
| 来源桶 | `source` | 左 | `text-muted-foreground`,**必须经规范化处理**——`scoop search` 有时返回 manifest 完整路径而非纯桶名,展示层只显示提取出的桶名,完整原始值放入 `title` 属性(悬停可见),与 P04/P07 的桶名呈现方式一致 |
| 状态 | (派生,非数据字段) | 左 | 见第 6 节 |
| 操作 | (派生) | 右 | 见第 6 节 |

**表格不包含 `binaries`(可执行文件)列**——`SearchResult` 数据结构中存在该字段,但本页不展示;可执行文件清单属于"详情"层级信息,只在 P07 详情对话框呈现(`scoop info` 的 `Binaries` 字段),避免列表页信息过载。

行为规则:

- 整行可点击(`cursor-pointer`),点击进入 P07 详情对话框(F04 出口);操作列的安装按钮需要阻止事件冒泡,避免点击安装按钮时同时触发行点击打开详情。
- 行 hover 态:`hover:bg-secondary`(§8.6 通用规则)。本页**不使用**§8.6"选中行"高亮样式(`bg-primary/10` + 左侧色条)——点击行为是"立即打开详情对话框"而非"停留在列表内选中",不存在需要持久标记的"当前选中行"状态,该样式更适用于 P08 任务列表这类"列表 + 详情区并存"的布局。
- 表格外层 `TableWrap` 只允许纵向滚动,不出现横向滚动条(项目级约束);列超宽内容用 `text-ellipsis` 截断,完整值放 `title`。
- **键盘可达性(设计目标)**:表格行应可通过键盘聚焦并以 `Enter`/`Space` 激活,确保纯键盘用户也能触发"查看详情"。**现状说明**见第 14 节第 3 条——这是面向未来实现的设计目标,当前代码尚未落地。

## 6. 状态列与操作列(安装入口)设计

这两列是本页与 P04 最大的差异点,因为 P05 的软件包只有"未装 / 安装中 / 已装"三种简化呈现(不显示"过期"——过期比对只对已装软件包有意义,属于 P04 语义):

| 软件包状态 | 状态列 | 操作列 |
|---|---|---|
| 可装未装(默认) | 空 | `sm` 尺寸主按钮,`Download` 图标 + "安装" |
| 安装中(该包名存在一条 queued/running 的 InstallJob) | `info` variant Badge,`LoaderCircle` 旋转图标 + 当前任务类型文案(如"安装") | 同一个安装按钮保留渲染,但置为 `disabled`(不隐藏按钮,只置灰,避免布局跳动,符合"按钮禁用"硬约束) |
| 已装(任意版本) | `success` variant Badge,`Check` 图标 + "已安装" | 空(不重复渲染安装入口) |

设计要点:

- **本页判断"已装"只看"是否已安装",不看版本新旧**——即使该软件包在 P04 中被标记为"过期",本页仍统一显示"已安装"徽标,不显示"过期"徽标、也不提供"更新"入口(更新操作属于 P04 的业务范围,详见 `flows/scoop-gui.md` §1.5)。这一列只是让用户"知道不用再点安装",不是重复 P04 的过期管理功能。
- 安装按钮**没有二次确认弹窗**——点击即刻发起(F05),这是与 P04"卸载"按钮(destructive,必须二次确认)刻意的反差设计:安装是可逆的增量操作(装错了可以卸载),卸载是有代价的破坏性操作,二者的确认成本应该不同。反馈完全依赖 F17:Toast(§8.10,4 秒自动消失的成功提示 / 6 秒的失败提示)+ P08 任务面板(点击安装后自动展开)。
- 按钮尺寸统一用 `sm`(32px 高),对齐表格 40px 行高的视觉密度,不用默认 40px 高按钮撑大行高。

## 7. 空状态与错误状态设计(F19)

两种状态共用 P10 定义的 `EmptyState` 呈现方式,只替换"表格所在区域"(搜索框不受影响,持续可见,见第 4 节):

| 场景 | 图标 | 图标容器配色 | 标题 | 提示文案 | 操作 |
|---|---|---|---|---|---|
| 搜索无匹配 / 可装范围为空 | `Package`(默认) | `border-border` + `bg-secondary` + `--fg-subtle` | `browse.empty` = "没有匹配的软件包" | `browse.emptyHint` = "换个关键字,或到"桶管理"添加更多桶" | 无独立按钮——用户直接在仍然可见的搜索框里清空/修改关键字即可,不重复放置"清空"按钮 |
| 读取失败(本机 Scoop 不可用等) | `TriangleAlert`(error 态) | `border-warning/30` + `bg-warning`(低透明度) + `text-warning` | `browse.loadFailed` = "可装软件包列表读取失败" | `error.scoopUnavailable` = "本机 Scoop 当前不可用,相关数据无法读取。请检查 scoop 命令是否正常后刷新重试。" | `secondary` variant"刷新"按钮,复用 PageHeader 同款刷新动作 |

两种场景共用同一套空态视觉容器(不因"真的没有数据"与"搜索没匹配到"切换不同插画或版式),差异只体现在文案与是否带重试按钮——刻意的克制设计,避免为一个列表页的空态维护过多视觉分支。

错误态**不**使用 Toast 或 Dialog,字面上与 §9 硬约束"错误展示用 Toast 或 Dialog,不用 inline alert"有出入,但本页的"读取失败"属于**整块数据缺失**场景,对应模块 PRD 中独立定义的 P10 子状态呈现,与 §9 硬约束真正约束的对象——"一次性命令/操作类错误"(如安装失败)——是不同层级的错误;操作类错误在本页仍然严格通过 Toast + P08 呈现,不冲突。此说明只为避免歧义,不视为对设计系统的偏离。

## 8. 加载状态设计

- 触发时机:首次进入本页(`availableLoaded` 为假)、或用户点击刷新按钮强制重新加载。
- 呈现:替换表格区域(搜索框保留),纵向居中堆叠:`LoaderCircle` `size-8` 旋转图标(`text-primary`)+ 下方文案 `browse.loadingAll` = "正在读取全部可装软件包(首次较慢,取决于桶大小)…"。
- 该文案主动管理用户预期("首次较慢")——可装列表是一次性全量拉取(不分页、不分批),桶越大首次等待越明显,设计上用文案而非进度条管理预期(没有可计算的百分比进度,不适合用确定性进度条)。
- 加载期间搜索框不禁用(用户可提前输入,等数据到达后立即按当前输入内容过滤展示),刷新按钮禁用。

## 9. 列表统计与容量提示

Footer 只有一行文本(不做分页控件,呼应 §10.2"分页(预留)"——本阶段不实现真分页):

- 常态:`browse.showing` = "显示 {shown} / {total} 条"。
- 当匹配结果超过前端渲染上限(实现侧上限为 500 条)时,额外追加 `browse.narrowHint` = "· 输入关键字缩小范围"提示——这是一条纯前端性能保护措施(避免表格一次性渲染上千行导致卡顿),设计上只需要一行辅助文案,不需要额外的警示配色或图标,`text-xs` + `--fg-subtle` 足够。

## 10. 交互反馈与禁用态规范

| 操作 | 反馈 |
|---|---|
| 点击刷新 | 按钮立即禁用 + 图标切换为旋转态;数据返回后表格整体刷新 |
| 点击安装 | 按钮立即禁用(该行);状态列同步切换为"安装中"Badge;Toast 与 P08 面板承接后续进度(不在本页内嵌进度条) |
| 点击行(非操作列) | 打开 P07 详情对话框 |
| 输入搜索关键字 | 表格与统计文案同步更新,无中间态 |

所有禁用态一律 `disabled:opacity-50 disabled:cursor-not-allowed`(§9 硬约束),不使用自定义禁用配色。

## 11. 键盘与无障碍

- `Tab` 顺序:刷新按钮 → 搜索框 → 表格行(可聚焦)→ 每行安装按钮,与视觉顺序一致。
- 搜索框获得焦点后,`Enter` 键不触发任何提交动作(本页没有"提交搜索"这个动作,过滤是持续实时的,`Enter` 应无副作用)。
- `Ctrl+K`(§9"预留"约束):本页不单独实现,若未来在 P03 主界面壳层级统一实现全局 `Ctrl+K` 唤起搜索,应导航到本页并聚焦搜索框,具体实现属于 P03 范畴,本文档不展开。
- 表格行需要可通过键盘访问(`role`/`tabIndex` 或原生可聚焦元素),确保纯键盘用户也能触发"查看详情"。**现状说明**:`src/pages/BrowseView.tsx` 当前的表格行只绑定了鼠标 `onClick`,未设置 `tabIndex`/`role`/键盘事件,纯键盘用户目前无法从列表行打开 P07——这是本设计提示词记录的一处待改进项,不在本次 P05 文档产出范围内直接修复(详见第 14 节)。

## 13. 与其它页面的关系

- **入口**:P03 主界面壳侧边栏"浏览软件包"导航项。
- **出口 → P07**:点击表格行,展示该软件包详情与安装入口(与本页安装按钮功能重复,属设计上刻意的"快速入口 + 详情深入"两级操作,与 P04 对已装软件包的处理方式对称)。
- **出口 → P08**:点击安装按钮后,任务面板自动展开(`jobsPanelOpen: true`),呈现方式见设计系统 §10.4,本文档不重复展开。
- **数据前置依赖**:状态列的"已安装"判断依赖 P04 的已装列表数据(`installed`),该数据在 Boot Sequence 阶段随主界面一起加载,不由 P05 单独请求——这意味着理论上如果已装列表长时间未刷新,本页的"已安装"徽标可能出现极短暂的滞后,但由于每次安装成功后都会触发被操作软件包的单包刷新(架构决策 ARCH10),常规使用路径下不会出现可感知的不一致。

## 14. 与真实源码的核对说明

本设计提示词编写时对照了 `src/pages/BrowseView.tsx` 实际实现,以下几点是撰写前需要明确指出的关键事实(而非设计选择),供下游 developer / QA 核对:

1. **搜索不产生新的 IPC 调用**:`scoop_search` 在页面首次加载时以无参数形式调用一次(全量),之后所有"搜索"都是前端对已加载数组的 `.filter()`,与 API 契约中 `scoop_search(query?)` 支持按关键字查询的能力**不冲突但未使用**——这是页面实现层面的取舍,不是 API 契约的限制。
2. **状态列与操作列没有"过期"呈现**:即使某已装软件包在 P04 中被标记为过期,本页仍然只显示"已安装",不显示过期或提供更新入口。
3. **表格行目前是纯鼠标交互**:`onClick` 直接绑定在 `<TableRow>` 上,没有 `tabIndex`/`role="button"`/键盘事件处理,键盘用户无法仅凭 `Tab`/`Enter` 打开 P07 详情;第 11 节的键盘可达性描述是面向未来实现的设计目标,不代表当前代码已具备该能力。
4. **安装没有二次确认**,与卸载(P04)的确认弹窗形成对比,详见第 6 节。
5. **表格不展示 `binaries` 字段**,尽管它是 `SearchResult` 数据结构的一部分。
