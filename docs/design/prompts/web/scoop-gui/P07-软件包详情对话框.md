## 1. 设计目标

P07 是本模块唯一的"软件包详情"呈现面,不独立成页,而是叠加在 P04(已装列表)/ P05(浏览搜索)之上的居中对话框(挂载于 `App.tsx` 根节点,由全局状态 `detailName` 控制显隐,任意时刻至多一个实例)。设计需要:

- 用同一套骨架承载三种业务身份——**可装未装 / 已装最新 / 已装过期**——只靠头部 Badge 与底部按钮组表达差异,不引入额外布局分支;
- 在不离开来源列表上下文的前提下完成"看详情 → 做决定 → 发起操作"的最短路径;
- 贯彻设计系统 v2"静默高级 + 科技感":玻璃感对话框、run-green 微光、JetBrains Mono 版本号与等宽字段对齐,呼应 Scoop CLI 的终端原生感。

## 2. 布局与容器规格(设计系统 §8.4 / §10.3)

| 属性 | 规格 |
|---|---|
| 定位 | 视口居中,固定定位 |
| 宽度 | `min(672px, 100vw − 64px)`(即设计系统 `max-w-2xl`) |
| 最大高度 | `100vh − 96px`;超出时仅中部字段区滚动,头部与底部固定不动 |
| 圆角 | `--radius-xl`(16px) |
| 背景 / 边框 / 阴影 | `--bg-elevated` + `1px solid --border` + `--shadow-xl` |
| 遮罩 | `bg-black/60` + `backdrop-blur-sm`(约 4px 模糊) |
| 右上角关闭 | 32×32 圆角按钮 + X 图标,默认无背景,悬浮态 `--bg-overlay` |
| 开关动效 | 遮罩淡入淡出;内容淡入 + 轻微缩放(0.96→1),≤150ms,呼应设计系统"即时反馈"原则 |

对话框本身**不叠加** `.bg-grid` / `.bg-glow-top`(那是列表页顶部 1/3 区域的专属科技感层;设计系统 §9 硬约束列出的适用页面清单是 P03/P04/P05/P06/P08/P09,不含 P07)。对话框自身的科技感只体现在图标框的主色微光边框与主操作按钮的辉光上,保持弹层应有的克制,不与承载它的列表页抢视觉焦点。

## 3. 头部区

从左到右:图标框 → (名称 + 版本号 + 状态 Badge)。

| 元素 | 规格 |
|---|---|
| 容器 | `padding: 20px 24px 12px`,水平排列、垂直居中,内部元素间距 12px |
| 图标框 | 44×44,圆角 `--radius-md`,边框 1px 主色微光,背景为主色微光的柔化版,图标色 `--primary`;内含"包裹"图标(22px) |
| 名称 | 等宽字体(`--font-mono`),18px,字重 600;取自触发详情时传入的包名 |
| 版本号 | 等宽字体,13px,`--fg-muted`;取自字段数组中 `Version` 键的值;读取中或读取失败时留空不占位 |
| 状态 Badge | 二选一,详见第 6 节状态矩阵;可装未装态不展示 Badge,此时名称与版本号右侧无内容 |

名称与版本号必须同排且不因 Badge 挤压而截断——`min-width:0` + `flex:1` 保证与 Badge 共享空间时优先保留名称完整可读,必要时允许该行整体换行到第二行(而非省略号截断包名)。

## 4. 字段展示区(中部)

字段区渲染的是"字段名/字段值"二元组数组,而非固定字段模板——**字段集合随软件包而异**(取决于 scoop 该软件包 manifest 实际声明了哪些属性),设计必须按"标签 + 值"通用两列布局承载任意字段集合,不能为具体字段单独设计专属位置或数量上限。

| 呈现态 | 规格 |
|---|---|
| 读取中 | 区域居中,`--fg-muted` 文案 + 主色旋转加载图标,上下留白各约 32px |
| 读取失败 | 左对齐,警告色图标 + "详情读取失败"文案 + 原始错误文本(等宽、12px、允许换行与选中) |
| 正常 | 两列网格:标签列定宽 120px(`--fg-subtle`),值列自适应换行;行距 8px,列距 16px,字号 13px |

值列渲染规则按字段名逐条判定,而非整体统一样式:

1. **默认**:等宽字体,保留原始换行,允许用户选中复制;
2. **`Description` / `Notes` 两个字段例外**:改用正文字体(非等宽),因为内容是自然语言说明而非结构化数据;
3. **`Website` 字段**:渲染为超链接,`--info` 配色,悬浮下划线,新标签页打开;
4. **`Source` / `Bucket` 字段**:展示值需经"桶名规范化"处理(从 manifest 完整路径中截取桶名段,如 `…\buckets\main\bucket\x.json` → 仅展示 `main`),原始完整值保留在鼠标悬浮提示中,不丢弃;
5. **标签文案**:每个字段名先查双语字典,命中则显示对应译文;**未命中的字段名原样显示其英文原文**,不得报错、不得留空、也不得整行隐藏——这是应对"字段集合随包而异、字典无法穷举"的既定兜底策略。

字段区自身最小高度 120px,超出对话框剩余高度时内部单独滚动,不影响头部与底部的固定可见性。

## 5. 底部操作区

| 元素 | 规格 |
|---|---|
| 容器 | 顶部 1px 分隔线,`padding: 16px 24px`,内容整体右对齐,按钮间距 8px |
| "关闭" | 次要(secondary)按钮,常驻显示,与业务状态无关,任何时候可点击 |
| 状态相关按钮 | 见第 6 节状态矩阵,紧随"关闭"之后追加 |

"关闭"按钮是本对话框**唯一不随状态变化的元素**——无论读取中 / 读取失败 / 忙碌 / 任意业务身份,用户必须始终有一条不产生副作用的退出路径。

## 6. 状态矩阵(业务身份 × 是否忙碌 → 底部按钮)

业务身份由三个独立信号推导:该包是否在本机已装清单中(`isInstalled`)、是否被判定过期(`isOutdated`)、该包名当前是否有排队中/执行中的任务(`busy`,任一写操作种类命中即视为忙碌,不区分具体是安装/卸载/更新中的哪一种)。

| isInstalled | isOutdated | busy | 头部 Badge | 底部按钮(紧随"关闭"之后) |
|---|---|---|---|---|
| 否 | — | 否 | 无 | 安装(primary) |
| 是 | 否 | 否 | 成功态"已安装" | 卸载(destructive) |
| 是 | 是 | 否 | 警告态"过期" | 卸载(destructive) + 更新(primary) |
| 任意 | 任意 | 是 | 与非忙碌时相同(不因忙碌而改变) | 整体替换为文字态:旋转图标 + "加载中…"(无可点击按钮) |

**忙碌态的特殊性**:Badge 不受忙碌影响,仍按 `isInstalled`/`isOutdated` 正常展示;只有底部的操作按钮被整体替换为不可交互的加载提示。也就是说"这个包现在能不能被发起新操作"与"这个包现在处于什么业务状态"是两件独立的事,设计上不要把二者耦合进同一套图形语言(例如不应该在忙碌时连 Badge 也换成"处理中")。

**"安装/更新"与"卸载"的操作确认不对称**(设计必须保留此不对称,不得为追求视觉统一而抹平):

- 点击"安装"或"更新":**不出现二次确认**,直接发起任务并关闭本对话框,转入 P08 进度呈现;
- 点击"卸载":**关闭本对话框,并把来源上下文切换为"已装列表"(P04)**,由 P04 承载的二次确认对话框询问用户(标题与正文需明确写出目标包名),确认后才真正发起任务转入 P08;取消则停留在 P04,不产生任何任务。

这一不对称是刻意的破坏性操作保护(卸载是本页面唯一的破坏性动作,理应比安装/更新多一道确认),但也意味着"从 P05 打开一个已装包详情并点击卸载"时,用户会被带离 P05——设计与验收都需要显式确认这一跳转对用户是可理解的。

## 7. 交互与动效

- **打开来源**:P04 已装行 / P05 可装行的整行点击(不含行内已单独处理点击事件的操作按钮列);
- **关闭途径**(四选一,效果等价——只关闭对话框,不改变任何业务状态):右上角 X、底部"关闭"按钮、点击遮罩空白处、`Esc` 键;
- **按钮按下反馈**:按下时轻微缩小(约 0.98 倍)+ 轻微变暗,约 100ms 过渡,与设计系统 §8.1 通用按钮反馈一致;
- **主操作辉光**:安装/更新按钮使用主色辉光,悬浮时辉光增强;卸载按钮使用破坏色辉光;
- **加载指示**:读取中的字段区图标与忙碌态的按钮区图标共用同一旋转动画(线性、无限循环),不额外设计第二种加载图形。

## 8. 与设计系统 token 的对照

| 用途 | Token |
|---|---|
| 对话框背景 / 卡片背景 | `--bg-elevated` |
| 主文本 / 次文本 / 辅助文本 | `--fg` / `--fg-muted` / `--fg-subtle` |
| 默认边框 / 强调边框 | `--border` / `--border-strong` |
| 主色(安装/更新按钮、成功 Badge、图标框强调、Website 之外的强调) | `--primary` + `--primary-glow` |
| 警告(过期 Badge) | `--warning` |
| 破坏性(卸载按钮) | `--destructive` + `--destructive-glow` |
| 链接(Website 字段值) | `--info` |
| 圆角 | 对话框 `--radius-xl`;按钮/图标框 `--radius-md`;Badge `--radius-sm` |
| 阴影 | 对话框 `--shadow-xl` |

字体的使用是本页面唯一偏离"标题用 heading 字体"通用惯例之处,需特别强调:包名与版本号**均使用等宽字体**(呼应终端字符对齐感),而不是设计系统 §4 默认给标题(H1-H3)分配的 Space Grotesk;字段值默认同样使用等宽字体,仅 `Description`/`Notes` 两个字段改用正文字体(DM Sans)。按钮文案统一使用正文字体。

## 9. 可访问性

- 对话框语义应满足:`role="dialog"`、`aria-modal="true"`、可被 `aria-labelledby` 关联到包名标题、打开时捕获焦点、关闭后把焦点归还给触发它的行元素;
- 右上角关闭按钮必须带 `aria-label`(纯图标按钮不能只靠视觉传达用途);
- 键盘:`Esc` 关闭;`Tab` 顺序需匹配视觉顺序(图标关闭 → 字段区内可聚焦元素如 Website 链接 → 底部按钮从左到右);
- 聚焦环统一使用设计系统的主色聚焦环样式。

## 10. 与上游 PRD 的已知差异(仅登记,不回改上游)

设计与原型均以 `src/components/DetailDialog.tsx` 真实实现为准产出;以下两点上游表述与实现存在出入,designer 阶段照实呈现现状,不在本文件内修正上游契约:

1. **"重试"入口缺失**:页面 PRD 第 5 节验收要点写"详情读取失败时,用户应看到可理解的错误提示,并可选择重试或关闭",但实现的读取失败态只有错误文案 + 常驻"关闭"按钮,没有专门的"重试"控件(双语字典中 `common.retry` 词条已定义但未被本组件引用)。本设计提示词按现状只呈现"关闭"路径。
2. **"桶详情"不在本对话框范围内**:模块 PRD 第 3 节把 P07 描述为"展示单个软件包或单个桶的详细信息",但页面级 PRD、`flows/scoop-gui.md`、以及 `DetailDialog.tsx` 三者一致地只覆盖"软件包"(已装/可装),`detailName` 从未由桶管理页(P06)设置。本设计提示词按现状只设计软件包详情,不包含桶详情形态。
