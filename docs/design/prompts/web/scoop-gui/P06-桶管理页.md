## 1. 页面定位与设计目标

P06 服务于模块 PRD 的 F09(已添加桶列表查看)、F10(已知桶清单查看)、F11(桶添加)、F12(桶移除),横切 F17(任务进度反馈)、F19(错误处理)。它是"可装软件包来源范围"的可视化管理面板,设计目标:

- 让用户在同一页面看清"本机已经有哪些桶"与"Scoop 还知道哪些桶可以加",两者用 Tab 分区而非同页堆叠或左右分栏,避免信息过载(呼应 flow §1.7 与页面 PRD 第 3.1/3.2 节)。
- 添加与移除都是会触发本机 git 操作(clone / remove 仓库)的长时动作,设计上必须清楚呈现"进行中"状态,并让用户知道结果去哪里看(P08 任务面板),不能让用户误以为点击后"没反应"。
- 移除是相对有后果的动作(会影响 P05 可装范围,flow §1.7.2),添加是低风险动作(可再移除);两者在确认强度上应有差异:添加免确认直达,移除必须过二次确认且用 destructive 强调。

## 2. 页面骨架(继承 §10.1 主界面壳 + §10.2 列表页)

P06 不是独立窗口,是 P03 主界面壳内 `<main>` 承载的一个 View,骨架 = §10.1 壳 + §10.2 列表页组合:

```
┌ Sidebar 240px(§8.7,关于底部区域的对齐说明见第 11 节) ─┬ PageHeader(bg-grid + bg-glow-top,标题+描述+工具栏) ┐
│                                                        ├ Tabs(已添加 / 已知桶清单)                          │
│                                                        ├ TableWrap(表头 sticky + h-10 行)                   │
│                                                        └ (无独立 Footer / 分页控件)                          │
└────────────────────────────────────────────────────────────────────────────────────────────────────────────┘
```

页面容器 `flex h-full min-w-0 flex-col`;PageHeader 区 `px-7 pt-6 pb-4` 并相对定位叠加 `bg-grid`/`bg-glow-top`(§9 硬约束"网格背景"明确覆盖 P06);内容区 `px-7 pb-6` + `flex-1 min-h-0`,与 Tabs 组合实现"表格区可纵向滚动、页面本身不整体滚动、不出现横向滚动条"。

## 3. PageHeader 设计指令

- 标题:`t("buckets.title")`("桶管理"/"Buckets"),`text-2xl font-semibold`,字体 Space Grotesk。
- 描述:`t("buckets.desc")`("桶是软件包的来源仓库。这里管理本机已添加的桶,并可从 Scoop 已知桶清单中添加新桶。"),`text-[13px] text-muted-foreground`,紧贴标题下方。
- 工具栏(标题右侧,`gap-2`,从左到右恰好两个按钮,不多不少):
  1. **刷新**(`outline` 变体):默认图标 `RefreshCw`;`loading.buckets = true` 时替换为 `--primary` 色的 `LoaderCircle`(旋转)且按钮本身 `disabled`——**仅刷新按钮随 loading 禁用,"添加自定义桶"按钮不受影响**,允许用户在刷新进行中仍打开添加对话框。
  2. **添加自定义桶**(`default`/primary 变体,发光):图标 `Plus`;点击打开自定义桶 Dialog(第 7 节)。
- 标题区 `min-w-0 flex-1`,确保窄窗口下工具栏不被挤出视口(呼应 §9 最小窗口 1024×600)。

## 4. Tabs 结构设计指令(§8.8)

真实结构是双 Tab,不是折叠面板或左右分栏:

- `TabsList` 底边 `1px solid --border`;两个 `TabsTrigger`:
  - **已添加**:文案 `t("buckets.addedTab")` + 一个 `text-muted-foreground` 的计数 `({buckets.length})`——**计数只出现在"已添加"上,"已知桶清单"不带数字**。
  - **已知桶清单**:文案 `t("buckets.knownTab")`,不带计数。
  - 默认激活 Tab = **已添加**(不是已知桶清单)。
  - 激活态:文字变 `--primary`,底边 `2px solid --primary`,叠加微光 `box-shadow: 0 1px 8px var(--primary-glow)`(§8.8 原文);未激活 hover 时文字变 `--foreground`,不出现底边。
- Tab 切换只做本地状态切换(不重新拉取数据),两个 Tab 的数据来自同一次 `refreshBuckets()` 请求(`Promise.all([bucket_list, bucket_known])`),已在内存中。

## 5. "已添加"表格设计指令(F09 · §8.6)

列(严格对应 `BucketInfo { name, source, updated, manifests }`,不得增删列):

| 列头 key | 中文/英文 | 对齐 | 内容 |
| --- | --- | --- | --- |
| `common.name` | 名称/Name | 左 | `Layers` 图标(`text-primary`)+ `font-mono font-medium` 桶名 |
| `buckets.repoUrl` | 仓库地址/Repository URL | 左 | `font-mono text-xs text-muted-foreground`,超宽截断(`max-w-[240px]`)+ `title` 属性给出完整地址 |
| `common.updated` | 更新时间/Updated | 左 | `text-xs text-muted-foreground` |
| `buckets.manifests` | 软件包数/Manifests | 右 | `font-mono text-muted-foreground` |
| `common.actions` | 操作/Actions | 右 | 见下 |

"操作"列是状态与动作的复用位——**没有独立的"状态"列**:

- 该桶正在被移除(`busy.has(name)`):渲染 `Badge variant="info"` + 旋转 `LoaderCircle` + 文案 `t("jobs.kind.bucket-remove")`("移除桶"/"Remove bucket"——**不是"移除中"这种进行时文案,字典里没有这个变体**,直接复用任务种类名)。
- 否则:`ghost` + `sm` 按钮,文字与图标 `text-destructive`,图标 `Trash2`,文案 `t("common.remove")`。点击打开第 8 节移除确认对话框,**不直接执行**。

表格容器 `TableWrap`:`overflow-y-auto overflow-x-hidden`,圆角 `--radius-lg`,边框 `--border`;表头 `sticky top-0 bg-card`;行 `h-10`,底部分隔线 `border-b border-border`,末行去掉;hover 背景 `bg-secondary`。**不出现横向滚动条**(项目级约束,超宽内容一律 ellipsis + title 提示)。

## 6. "已知桶清单"表格设计指令(F10/F11 · §8.6)

列:

| 列头 key | 中文/英文 | 对齐 | 内容 |
| --- | --- | --- | --- |
| `common.name` | 名称/Name | 左 | `Layers` 图标(已添加 `text-primary` / 未添加 `text-subtle`)+ `font-mono font-medium` 桶名 |
| `common.status` | 状态/Status | 左 | 见下 |
| `common.actions` | 操作/Actions | 右 | 见下 |

"状态"列:

- 正在添加(`busy.has(name)`):`Badge variant="info"` + 旋转 `LoaderCircle` + `t("jobs.kind.bucket-add")`("添加桶"/"Add bucket")。
- 已添加(该名字出现在"已添加"数组中):`Badge variant="success"` + `Check` 图标 + `t("buckets.alreadyAdded")`("已添加"/"Added")。
- 未添加且不在处理中:留空,不渲染任何徽标。

"操作"列:

- 已添加:**不渲染任何按钮**(留空),已添加的桶只能去"已添加" Tab 移除,不在本 Tab 重复提供操作入口。
- 未添加:`default` + `sm` 按钮,图标 `Plus`,文案 `t("common.add")`,`disabled = busy.has(name)`。**注意:添加进行中时该按钮依然渲染,只是置灰禁用,并不隐藏**——与"状态"列的进度徽标同屏共存,这是与"已添加"表格操作列(徽标/按钮二选一互斥渲染)不同的地方,设计走查时不要把这个共存态当成需要"优化掉"的冗余。
- 点击"添加"**不弹确认对话框**,直接入队(与移除的强确认形成对比,见第 1 节的风险分级)。

## 7. 添加自定义桶对话框设计指令(F11 · §8.4)

由头部工具栏"添加自定义桶"按钮触发,`Dialog` 宽度 `w-[min(480px,calc(100vw-64px))]`(比 §8.4 默认 `max-w-2xl` 更窄的紧凑表单尺寸),圆角 `--radius-xl`,遮罩 `bg-black/60 backdrop-blur-sm`:

- `DialogTitle` = `t("buckets.addCustom")`("添加自定义桶"/"Add custom bucket"),与触发按钮同一文案。
- 表单体(纵向 `gap-3`):
  1. **桶名称**(Label = `t("buckets.bucketName")`):`Input` `font-mono`,占位符字面量 `extras`(示例值,不经 i18n)。
  2. **仓库地址 `(可选)`**(Label = `t("buckets.repoUrl") + t("common.optional")`):`Input` `font-mono`,占位符 `https://github.com/user/scoop-bucket`;下方 `text-xs text-subtle` 提示 `t("buckets.repoHelp")`("已知桶无需填写;自定义桶需提供 git 仓库地址")。
  3. 两个输入框都监听 `Enter` 直接提交(与"添加"按钮等效)。
- `DialogFooter`:`secondary` "取消" + primary "添加"(`t("common.add")`),**"添加"按钮在桶名称为空时禁用**,仓库地址允许留空。
- 提交后:对话框关闭、两个输入框清空,`enqueue("bucket-add", 名称, 仓库地址||undefined)`,**不做名称格式的前端预校验**——`NAME_RE`/`REPO_RE` 校验在 Rust 侧(`docs/architecture/api/scoop-gui.md` §2),前端把输入原样提交,校验失败时的呈现见第 9 节"任务创建本身失败"。
- 取消 / 点击遮罩 / 点右上角 X 关闭时**不清空**已输入内容(只有成功提交才清空)——这是真实代码的行为,重开对话框应看到上次残留的输入值。

## 8. 移除桶确认对话框设计指令(F12)

复用跨页通用的 `ConfirmDialog`(卸载确认 / 批量更新确认共用同一组件),宽度同上 `min(480px, ...)`:

- `title` = `t("buckets.removeTitle")`("移除桶确认"/"Confirm bucket removal")。
- `message` = `t("buckets.removeDesc", { name })`("移除桶 {name} 后,该桶内未安装的软件包将不再出现在浏览列表;已安装的软件包不受影响。")——**必须原样告知对 P05 可装范围的影响面,不能只写"确定移除吗"这类空泛文案**,这是本对话框与其它确认框在文案深度上的差异点。
- `destructive = true`:确认按钮走 `destructive` 变体(红色 + 微光),取消按钮 `secondary`(灰,**不是** `cancel` 变体——`cancel` 变体专用于 P08 里"取消正在执行的任务",与本对话框"取消这次弹窗"是两个不同语义,不要混用);二者右对齐,取消在左、确认在右。
- `Enter` 键在对话框内直接等效点击确认按钮。

## 9. 任务反馈与状态语义(F17 · §8.9/§8.10)

- 点击"添加"(已知 Tab 一键添加,或自定义桶对话框提交)或确认"移除"后,**立即打开 P08 任务面板**(不是等任务真正开始跑才开,是入队成功那一刻就开面板),让用户第一时间看到"排队中"状态,不需要额外点击才能找到进度入口。
- 该 target 对应行随即从"可操作"态切换为"处理中"态(第 5/6 节的 Badge/禁用规则),两者是同一进行中语义的两处呈现——**页面级"操作中禁用"的粒度是"单个目标行",不是整页封锁**;同一页面里未处理中的其它行按钮保持可交互,"刷新"与"添加自定义桶"也不受影响。
- 成功:P08 侧 toast(`jobs.succeededToast`,默认 4s 自动消失,左侧 2px `--primary` 边线)+ 本页两个 Tab 的数据整体重新拉取(`refreshBuckets()` 重新调用 `bucket_list`/`bucket_known`,不是本地乐观更新单行)+ 若 P05 此前已经加载过可装列表,一并强制刷新(跨页一致性,呼应 flow §1.7 业务结果)。
- 失败:toast 改为 `jobs.failedToast`("{title} 失败,可在任务面板查看日志后重试",6s,`--destructive` 左边线)+ **强制重新打开 P08 面板**(即使用户中途关闭过),失败原因与完整日志只在 P08 呈现,**P06 本页不维护"添加失败"/"移除失败"的行内徽标**——失败后行回到失败前的可操作态(未添加 / 已添加保持不变),"重试入口"就是同一个按钮再点一次,不是另一个专门的"重试"按钮。
- 取消:toast 改为 `jobs.cancelledToast`(信息色,不强制开面板),行为与失败相同——回到动作前稳态,不进入任何"失败/取消"专属视觉态。
- 任务创建本身失败(如非法字符被 Rust 侧正则拒绝,`enqueue_job` 直接 reject):不产生任何 InstallJob,P08 不会新增条目,只有一条 `error.jobStartFailed` toast("任务创建失败:{msg}")。这是与"任务跑完后失败"完全不同的另一条错误路径,设计与验收都要覆盖到。

## 10. 空态与错误态设计指令(F19 · P10)

三种子状态,视觉形态不同,不要混用:

1. **本页级读取失败**(`bucket_list`/`bucket_known` 任一请求失败,两者共用同一次 `Promise.all`):Tabs 整体不渲染,替换为单一 `EmptyState(error)`——`warning` 色系图标框(`TriangleAlert`)、标题 `t("buckets.loadFailed")`("桶列表读取失败")、提示 `t("error.scoopUnavailable")`、下方带 `secondary` "刷新"按钮。**这是唯一带重试按钮的空态**,且**同时覆盖"已添加"和"已知"两部分的读取失败**——不要设计成两个 Tab 各自独立报错。此时 PageHeader 上原有的"刷新"按钮依旧可见并可点(两处刷新入口同时存在,调用同一个 `refreshBuckets()`)。
2. **"已添加"为空**(请求成功但本机确实没添加过桶):只在"已添加" Tab 内容区渲染 `EmptyState`(`Layers` 图标,中性灰),标题 `t("buckets.empty")`("本机还没有添加任何桶")。**不带 hint、不带跳转"已知" Tab 的按钮**——第 11 节对此与页面 PRD 验收要点的出入有说明,设计交付按代码现状(纯文案)执行,不要自行加回引导按钮。
3. **"已知"为空**(请求成功但 `bucket_known` 返回空数组):只在"已知桶清单" Tab 内容区渲染 `EmptyState`(`Layers` 图标),标题 `t("buckets.knownEmpty")`("已知桶清单读取失败或为空")。同样不带 hint/按钮。这条文案字面上提了"读取失败",但**只要走到这个分支就已经排除了请求失败的可能**(失败会被状态 1 的顶层错误态拦截),实际只会在"确实为空"时出现,文案略宽泛,不是功能缺陷。

## 11. 与真实源码的对齐说明(供下游核对,不回改上游契约文件)

以下几点是撰写本提示词与原型过程中,页面 PRD / 设计系统示意图与 `src/pages/BucketsView.tsx` 实际实现之间发现的出入。按任务要求,**均以代码为准**,本节只做记录:

1. **Sidebar 底部控件**:`web.md` §8.7/§10.1 的示意图画了"底部 Theme toggle(亮/暗/跟随系统)"与"[中文/EN]"两个常驻控件,但 `App.tsx` 的 `Shell()` 实际只在侧边栏底部放了一个"任务"导航项(带进行中任务数徽标),主题与语言切换实际收在 P09 设置页内,侧边栏本身不放这两个控件。原型按代码现状绘制侧边栏,不臆造设计系统示意图里的控件。
2. **"已添加为空"/"已知为空"空态没有引导性 CTA**:页面 PRD 第 5 节验收要点写"已添加桶列表为空时,应显示 P10 空状态并引导用户前往已知区段添加桶""已知桶清单读取失败时,应显示 P10 错误状态并允许用户重试"——但 `BucketsView.tsx` 对这两个分支调用 `EmptyState` 时都只传了 `icon`/`title`,**没有传 `hint` 或 `children`**,即没有"去已知区段看看"的引导按钮,也没有独立于顶层错误态之外的"重试"按钮。本提示词与原型按代码现状呈现(纯图标+标题),验收文档第 9 节把这一条列为待复核项。
3. **"已知读取失败"与"已添加读取失败"实际合并成一个错误态**:两个数据源在 `refreshBuckets()` 里共用同一个 `Promise.all`,任一失败都会让整份 Tabs 被同一个顶层 `EmptyState(error)` 替换,不存在"只有已知桶清单报错、已添加清单正常展示"这种分裂呈现。设计与验收都按"二者共用一套错误态"处理,不单独为"已知"设计独立的错误态。
