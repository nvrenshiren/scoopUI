# P10 · 设计提示词 · 空状态与错误页

> 文档层级:页面级产出(designer · 仅登记不送审)
> 适用范围:web 端 scoop-gui 模块 P10
> 上游依赖:`docs/prd/pages/web/scoop-gui/P10-空状态与错误页.md`、`docs/design/systems/web.md`
> 下游引用:`docs/design/prototypes/web/scoop-gui/P10-空状态与错误页.html`
> 真相源声明:本页面**没有独立组件**,其视觉形态已经在 `src/components/EmptyState.tsx` 与 `src/pages/InstalledView.tsx` / `BrowseView.tsx` / `BucketsView.tsx` 中实现。本提示词以这些真实源码为准绳描述"应当如何设计",而不是从零构思;如与页面 PRD 描述冲突,以代码为准(冲突点见第 8 节)。

## 1. 定位:这是一份"子状态"提示词,不是独立页面提示词

P10 不对应任何独立路由或组件。它描述的是 P03(主界面壳)、P04(已装列表)、P05(浏览/搜索)、P06(桶管理)在"数据为空"或"数据读不到"时,**内容区域整体替换**成的一种呈现形态。设计时必须始终带着宿主页面的头部(`PageHeader` + `bg-grid`/`bg-glow-top`)一起考虑,不能孤立设计成一张独立大页。

结构关系(自顶向下,均已在真实代码中固定):

```
[ 宿主页头部:标题 + 描述 + 工具栏按钮 ]   ← 不受子状态影响,始终渲染
[ 宿主页工具条:搜索框 / 过滤框(若该页有) ] ← 大多数情况下也不受子状态影响
[ 内容插槽 ]
  ├─ 有数据 → 表格
  ├─ 空(非错误) → EmptyState(中性灰)
  └─ 读取失败 / Scoop 不可用 → EmptyState(警示态)
```

唯一的例外是"读取失败"态:此时 P06 的 Tabs(已添加/已知桶清单)整体被替换,搜索框/工具栏按钮仍然保留在原位(详见第 6 节场景 5/6)。

## 2. 视觉基准:空状态容器规格(源自 `EmptyState.tsx`)

组件对外只暴露 5 个参数:`icon?`、`title`、`hint?`、`error?`、`children?`。设计一律在这 5 个参数的表达能力内完成,不引入组件不支持的形态(例如多张插画拼接、渐进式引导步骤条、专属配色的自定义卡片背景)。

```
┌───────────────────────────────────────┐
│                                       │  ← 容器:flex-1 居中(水平+垂直),
│              ┌───────┐               │     padding 24px(横) / 48px(纵)
│              │ Glyph │               │  ← 图标方框 56×56(size-14),
│              └───────┘               │     圆角 --radius-lg,1px 边框
│                                       │
│           标题文本(15px 中粗)          │  ← mb-1 与图标间距 4px
│                                       │
│      辅助说明(13px,最大宽 480px)       │  ← 可选,居中换行
│                                       │
│           [ 可选操作按钮 ]              │  ← 可选,mt-3
│                                       │
└───────────────────────────────────────┘
```

- 图标方框两套配色,**没有第三套**:
  - 中性(默认,`error` 未传或为 `false`):`border-border` 边框 + `bg-secondary`(即 `--bg-overlay`)背景 + `text-subtle` 图标色。
  - 警示(`error === true`):`border-warning/30` 边框 + `background: var(--warning-soft)`(琥珀色 5% 透明度)+ `text-warning` 图标色。
  - **不存在 destructive(红色)图标方框**——这是本页面最容易被误设计错的地方,详见第 5 节。
- 默认图标(未显式传 `icon` 时的回退):中性态回退为 `Package`(箱子);警示态回退为 `TriangleAlert`(警示三角)。三个宿主页只有 P06 的两个空列表分支显式传入了 `Layers`(桶形态)图标,其余全部使用回退图标,设计时不要为每个空状态都单独设计定制图标。
- 标题恒为纯文本,不加图标前缀、不加冒号;辅助说明是次要信息,可省略(`buckets.empty` / `buckets.knownEmpty` 两个分支就没有 `hint`,只有标题)。
- `children`(操作区)只在"读取失败"三个分支里出现,固定是一个 `variant="secondary"` 的"刷新"按钮;**其余空状态分支目前都没有可点击的操作区**,下一步动作只以 `hint` 文案的形式提示(见第 6 节逐场景说明与第 8 节讨论)。

## 3. 图标与"插画"设计指令

- **禁止引入插画风格元素**:不使用 undraw 风格的场景插画、不使用等距(isometric)图形、不使用多色渐变吉祥物图形。真实实现从始至终只用 `lucide-react` 的单色线性图标(2px 描边,24×24 视框),装在一个纯色方框里。设计提示词以"线性图标 + 方框底"为唯一基准。
- 图标来源限定在 lucide 图标集中已在本产品出现过的几个:`Package`(箱子,已装/可装默认)、`TriangleAlert`(警示三角,错误默认)、`Layers`(叠层,桶)。若后续新增场景需要新图标,应从 lucide 同风格线性图标中选取,保持 2px 描边、圆角端点(`stroke-linecap: round`)。
- 图标方框尺寸固定 56×56px(`size-14`),圆角 `--radius-lg`(12px),图标本体 28×28px(`size-7`),不随场景放大缩小——不要为了突出"更严重"的错误而放大图标或加粗描边,严重度只通过第 5 节的配色分级表达,不通过尺寸。

## 4. 文案层级设计指令

三级文案层级,严格对应组件的三个文本 slot,不得合并或增加层级:

1. **标题(title,必填)**:一句话陈述"当前是什么状态",不解释原因、不给建议。例如"本机还没有通过 Scoop 安装任何软件包""可装软件包列表读取失败"。字号 15px、中等字重、`--fg` 主文本色(warning 态标题颜色不变,仍是 `--fg`,只有图标方框变色——不要把标题文字也染成警示色,那样会在暗色背景上产生额外的视觉噪音)。
2. **辅助说明(hint,可选)**:解释"接下来能做什么"或补充一句业务背景,13px、`--fg-muted`(`text-muted-foreground`)、最大宽度 480px 居中换行。这一层**不写底层报错信息**(不写 stderr、不写异常堆栈、不写命令退出码)——真实代码里 `errors.*` state 确实存了原始报错字符串,但从未渲染在这一层,只用于内部诊断。设计时延续这个原则:hint 永远是人话,不是技术日志。
3. **操作按钮(children,可选)**:仅在"读取失败"三态出现,文案固定为"刷新"(`common.refresh`),`variant="secondary"`,前缀 `RefreshCw` 图标。不要为"空(非错误)"状态凭空加操作按钮——那是真实代码里没有的变体(参见第 8 节的设计侧建议,那是"未来可做"而不是"现在应该做")。

双语:所有文案必须同时有 zh/en 两版,经 `t(key)` 取值,不得出现半中半英;不得为 P10 新造未在 `src/i18n.ts` 里注册的 key——本设计提示词覆盖的文案全部已存在于 `error.*` / `installed.empty*` / `browse.empty*` / `browse.loadFailed` / `buckets.empty` / `buckets.knownEmpty` / `buckets.loadFailed` 这些既有 key 下,不新增字典条目。

## 5. 错误态展示形式设计指令(硬约束落地 + 两级错误视觉语言)

设计系统 §9 硬约束:"错误用 Sonner Toast 或 Dialog;不用 inline alert;错误码用 font-mono"。P10 范围内,这条约束具体落地为**两级、两种配色的错误语言**,设计时必须分清楚用在哪一级,不能混用:

| 级别 | 触发场景 | 承载形式 | 配色 | 是否显示原始报错 |
|---|---|---|---|---|
| **业务级(此级=P10 本页范围)** | 列表读取失败 / Scoop 运行期间不可用(场景 5/6) | `EmptyState` 整体替换内容区(`error` 属性) | **警示琥珀色**(`--warning` / `border-warning/30` / `bg-[var(--warning-soft)]`)——**不是红色** | 否,只给业务话术 |
| **任务级(此级=场景 7,不在 P10 组件范围但会回落到 P10 覆盖的来源页)** | 某条 InstallJob 执行失败 | Sonner Toast(`toast.error`,6 秒) + P08 任务面板里的该条任务行 | **destructive 红色**(Toast 左边线 `border-l-2 border-destructive`;任务面板图标/徽标 `text-destructive` / `variant="destructive"`) | 面板日志区用 `font-mono` 展示原始 stdout/stderr(真正落地"错误码 font-mono"这条约束的地方) |

设计上必须保留这个区分的原因:业务级是"环境暂时不可用,过会儿再试就好"的降级语气(琥珀色更平和),任务级是"你刚才那个操作确实失败了"的明确语气(红色更强烈)。如果把 P10 的读取失败态也设计成红色,会让"打开软件包列表"和"卸载失败了"在视觉上一样严重,削弱了红色在本产品里"操作失败/需要用户处理"的信号强度。

关于"不用 inline alert":EmptyState 是**整体替换内容区**(表格/列表原本占据的区域被完全替换),不是在保留原内容的同时于顶部/侧边插入一条 banner。这正是"不用 inline alert"应有的落地方式——判断标准不是"有没有一个方框",而是"周围是否还保留着本应被遮蔽的旧内容"。P01(启动检测页)当前用了一个保留表单内容、仅在其上方插入红色提示条的写法(`setup.failedDesc` 那块),那属于 P01 自己的实现,不是 P10 的范围,也不构成本设计提示词的参照对象。

## 6. 逐场景设计指令

对应页面 PRD 第 4 节的场景 1~7,每条给出:触发条件 → 真实代码取值 → 设计要点。

### 场景 1 · 已装列表为空(P04)
- 代码取值:`icon` 缺省(回退 `Package`,中性灰)、`title = t("installed.empty")`("本机还没有通过 Scoop 安装任何软件包")、`hint = t("installed.emptyHint")`("前往"浏览软件包"查找并安装第一个软件包")、无操作按钮。
- 设计要点:hint 里提到的"浏览软件包"是纯文本强调(无下划线、无颜色区分、不可点击),不要设计成一个可点击的链接态——那会与真实实现的静态文本产生视觉预期落差。真正的下一步操作入口是侧边栏的"浏览软件包"导航项,本状态只负责"提示去哪"。
- 补充事实:此分支同时承接"过滤后无匹配"(参见第 8 节 8.1)。

### 场景 2 · 可装列表为空(无查询,P05)与场景 3 · 搜索无结果(P05)
- 代码取值:两者**渲染路径完全相同**——`filtered.length === 0` 时一律 `icon` 缺省(回退 `Package`)、`title = t("browse.empty")`("没有匹配的软件包")、`hint = t("browse.emptyHint")`("换个关键字,或到"桶管理"添加更多桶"),无操作按钮。
- 设计要点:不要为这两个场景画两套不同的视觉稿。设计交付物里应当只有一个"P05 空结果"画面,通过旁注说明它同时对应"桶内本来就没有可装包"与"关键字没搜到"两种触发路径。搜索框本身在此状态下依然渲染在内容插槽上方,保持可编辑,不做禁用处理。

### 场景 4 · 桶列表为空(P06,已添加 / 已知桶清单两个 Tab 独立判断)
- 已添加 Tab 为空:`icon={Layers}`(唯一一处显式覆盖默认图标,中性灰底,**不是警示色**)、`title = t("buckets.empty")`("本机还没有添加任何桶")、无 `hint`、无操作按钮。
- 已知桶清单 Tab 为空:`icon={Layers}`(同样中性灰底)、`title = t("buckets.knownEmpty")`("已知桶清单读取失败或为空")、无 `hint`、无操作按钮。
- 设计要点:两个 Tab 各自独立判空,互不影响——已添加为空不代表已知清单也为空,反之亦然。两者视觉上都是"中性"配色,即使 `buckets.knownEmpty` 这条文案字面提到"读取失败",也不要因此把图标方框设计成警示色——它在代码里就是普通(非 `error`)分支,警示色留给场景 5 那个更外层的整体失败态。

### 场景 5 · 读取失败(P03/P04/P05/P06 任一区段)与场景 6 · 运行期间 Scoop 不可用
- 代码取值:三个宿主页各自的 `errors.*` 为真时,`icon` 缺省(回退 `TriangleAlert`,警示琥珀色)、`title` 因页面而异(`error.readFailed` "无法读取数据" / `browse.loadFailed` "可装软件包列表读取失败" / `buckets.loadFailed` "桶列表读取失败")、`hint` 三页统一复用 `error.scoopUnavailable`("本机 Scoop 当前不可用,相关数据无法读取。请检查 scoop 命令是否正常后刷新重试。")、操作区固定一个"刷新" `secondary` 按钮。
- 设计要点:场景 5(泛化的读取失败)与场景 6(运行期间 Scoop 整体不可用)在真实代码里**是同一个视觉分支**,不需要也不应该设计成两套画面——两者共用同一句 hint 文案,已经用"检查 scoop 命令是否正常"这种业务语言把两种根因都覆盖了,不应再引入区分。P06 触发此态时,替换的是整个 Tabs 区域(两个 Tab 一起消失),头部工具栏(刷新/添加自定义桶按钮)保留原位。

### 场景 7 · 长时任务失败(P08 结果回落到来源页)
- 这不是 `EmptyState` 的变体,而是三件事的组合,设计时必须按真实交互顺序还原,不能设计成一张"失败结果卡片"独立画面:
  1. 失败瞬间:右上角 Sonner Toast(红色左边线,6 秒自动消失),文案 `t("jobs.failedToast", { title })`。
  2. P08 任务面板(非 modal 浮层)里该任务行:红色 `TriangleAlert` 图标 + `destructive` 徽标"已失败(可重试)";点击该行展开下方日志区(`font-mono`,原始输出滚动到底)。**该行没有"重试"或"放弃"按钮**——面板对终态任务只保留可选中查看日志的能力,不提供操作按钮(取消按钮只在排队中/执行中出现)。
  3. 来源页(如 P04)对应行:因为该软件包的实体状态已经回退到失败前的稳态(如"已装过期"),行内原有的"更新"按钮自然重新可点击——这就是"重试"的真正入口,不是面板里的按钮。
- 设计要点:如果要为这个场景画原型/走查图,请画出这三个画面的组合(或至少用旁注标出三者关系),不要只画一张"任务失败详情页"——那种独立详情页在当前实现里不存在。

## 8. 设计侧已知局限(仅登记,不阻塞开发;不代表要求 developer 改代码)

以下几点是核对真实代码时发现的、页面 PRD 期望与当前实现之间的差距。按本任务要求,这里只登记、不修改 PRD、不反过来要求代码改动;是否收敛由后续迭代决定。

- **8.1 "空"与"过滤无结果"共用同一句文案**:P04 的 `installed.empty`("本机还没有通过 Scoop 安装任何软件包")在真实代码里同时用于两种情况——本机真的一个都没装,以及本机装了但过滤框关键字没匹配到。后一种情况下这句"还没有安装任何软件包"字面上并不准确(明明装了,只是没匹配上)。P05 也有类似的场景 2/3 合并现象(见第 6 节)。这是当前实现的既有行为,不是本次设计要修的东西,但设计交付物里不应该"过度设计"出一个当前不存在的"过滤无结果"专属文案分支,以免和实现脱节。
- **8.2 空状态普遍没有可点击的下一步操作**:除三个"读取失败"分支外,其余空状态都只有文字提示,没有按钮。页面 PRD 的验收要点写"每个空状态或错误状态都应提供至少一个明确下一步动作",目前只在"错误"分支满足(刷新按钮),"空"分支的"下一步动作"目前只体现为提示文字里点名的目的地,不是可点击控件。若未来要补齐,建议的落点是 `EmptyState` 的 `children` 插槽(该插槽已经存在,补按钮不需要改组件结构),但这属于开发侧的后续工作,不在本次设计提示词的产出范围内。
- **8.3 "已知桶清单读取失败或为空"这句文案在当前控制流下只可能因为"空"触发**:`refreshBuckets()` 把"已添加桶"和"已知桶清单"两个请求合并成一次 `Promise.all`,任一失败都会走场景 5/6 的整页失败态(替换掉整个 Tabs),因此场景 4b 这个分支被触发时,数据请求其实是成功的,只是返回了空数组——文案里"读取失败"四个字在这条路径下永远不会是真实原因。同样仅登记,不要求现在改字典。
