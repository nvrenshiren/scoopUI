## 1. 设计目标与基调

P01 是用户第一次看到本产品"内容区"的地方(P02 语言选择之后),也是唯一会**整页接管**、不显示 240px 侧边栏与主界面壳的页面——参照设计系统 §1 三条最相关原则来定调:

- **终端原生感**:用户此刻正在等待一个"类命令行"动作(`scoop --version` 探测 / `install.ps1` 执行),视觉上要让人联想到终端而非普通表单向导——检测提示与安装日志一律 `font-mono`,日志区直接复用终端配色(近黑底 + 等宽字体)。
- **静默高级**:这是一个高风险页面(用户可能第一次接触 Scoop,也可能是安装重试),不用大色块或强插画分散注意力;整页背景只叠加克制的 `.bg-grid` + `.bg-glow-top`,前景收敛为一张居中卡片。
- **即时反馈**:检测中 / 安装中两个耗时态都必须有明确的、持续的动效(旋转图标或扫动进度条),不能让用户怀疑"卡住了"。

设计基调:**克制、居中、单卡片、以 warning(amber)而非 primary(green)作为本页图标强调色**——因为本页的默认叙事是"还没准备好"(未检测到 Scoop / 安装中),`primary` 绿色留给"已经可用"之后的 P03 及其他常态页面。

## 2. 页面状态划分与骨架总览

页面对应 `flows/scoop-gui.md` §2.6 Boot Sequence 与 §2.2 Scoop 自身状态机,共呈现 3 组视觉状态(不新增状态名,只是视觉呈现层的分组):

| 视觉状态 | 对应 Boot Sequence / Scoop 自身状态 | 功能引用 |
|---|---|---|
| A. 检测中 | `检测 Scoop`(Scoop 自身 = 检测中) | F15 |
| B. 未检测到 Scoop → 安装配置表单 | `未检测到 Scoop`(Scoop 自身 = 未安装 / 不可用(安装失败)) | F16、F19 |
| C. 协助安装中 | `协助安装中`(Scoop 自身 = 协助安装中) | F16、F17 |

三态共享同一套背景骨架:

```
┌──────────────────────────────────────────────┐
│ .bg-grid(全屏)+ .bg-glow-top(顶部 40% 区域)  │
│                                                │
│              ┌──────────────────┐            │
│              │ 居中卡片 720px     │  ← 状态 A 时无卡片,仅居中图标+文案
│              │ (状态 B / C 呈现)  │
│              └──────────────────┘            │
└──────────────────────────────────────────────┘
```

- 状态 A 不使用卡片容器,直接在网格背景正中呈现一个"检测中"文字组合(见 §3)。
- 状态 B / C 共用同一张卡片外壳(§8.2 Card 规格:`bg-card` + `1px solid border` + `radius-lg` + `shadow-md`),卡片顶部带一条 1px 渐变线(`.gradient-line`,呼应侧边栏顶部品牌线的科技感语言),卡片内部内容随状态切换。
- 全页无侧边栏、无 PageHeader、无 sidebar 品牌线——这是本产品唯一一处"裸内容"页面,设计上应通过卡片本身的边框 + 渐变线 + 网格背景来承担"产品感",而不依赖外壳。

## 3. 状态 A · 检测中(F15)

- 布局:整个视口居中(水平垂直双居中),不使用卡片,只有一组纵向堆叠的元素:
  1. `LoaderCircle` 图标(或等价旋转图标),`size-8`,`text-primary`,持续旋转动画。
  2. 主文案(默认字重 medium,约 15px):检测中提示语。
  3. 副文案:`font-mono`、`text-xs`、`--fg-subtle` 色,呈现"正在执行的具体命令"这一类终端感的辅助信息——这是本页"终端原生感"最直接的落点。
- 不使用 Badge、不使用进度条(检测本身是一次性快速命令,不建模为可衡量进度)。
- 该状态没有任何可交互元素(无按钮、无取消入口)——检测动作耗时很短,不提供取消,避免过度设计。

---

## 4. 状态 B · 未检测到 Scoop / 安装配置表单(F16)

### 4.1 卡片头部

- 左侧:`size-12` 图标容器,圆角 `radius-lg`,边框 `border-warning/30`,背景为 warning 色 15% 透明度叠加,图标本身 `text-warning`——**用 warning(amber)而非 destructive(red)**,因为"未检测到 Scoop"是一个中性的首次状态,不是错误。
- 右侧:标题(小标题量级,§4 `text-xl`/1.25rem,`font-heading` Space Grotesk,通过 `<h2>` 语义标签即可继承标题字体)+ 一行说明正文(`text-[13px]`、`--fg-muted` 色),说明"本机没有 Scoop、将由本产品协助安装、需要联网"。

### 4.2 八项安装配置(F16 唯一合法字段集)

严格按以下顺序,一列纵向表单(单列,不做分栏网格,除代理凭据一组外):

1. **安装根目录(ScoopDir)**——文本输入 + 一行灰色说明。
2. **全局软件包目录(ScoopGlobalDir)**——文本输入,标签右侧带一枚 `warning` variant 的小 Badge("需要管理员权限"),提示这项修改的权限含义;说明文字要点出"默认路径 + 修改此项需要管理员权限"。
3. **下载缓存目录(ScoopCacheDir)**——文本输入 + 说明。
4. **代理服务器(Proxy)**——文本输入 + 说明(示例占位符用真实代理 URL 形态,如 `http://127.0.0.1:7890`)。
5. **代理例外(NoProxy)**——文本输入 + 说明(逗号分隔主机名)。
6. **代理凭据一组**——横向两列等宽:用户名(带说明文字"代理需要身份验证时才填")+ 密码(`type=password`,无需额外说明,与用户名对齐即可,不必重复解释)。
7. **使用 Windows 默认凭据访问代理**——单行 Checkbox + 文案,不带额外说明段落(标签本身已经把括号内的官方参数名带出)。
8. **以管理员权限运行安装(RunAsAdmin)**——Checkbox,文案下方跟一行说明(UAC 提权的后果 + 不勾选时的推荐路径)。

所有输入框统一 `font-mono`——本页所有路径 / URL / 主机名类字段都是"类终端参数",与设计系统 §4"代码/版本号/错误码用 JetBrains Mono"的原则呼应,即使设计系统没有单独为表单输入框规定字体,本页的字段语义(文件路径、代理地址)决定了它们应该被当作"代码值"处理。

**关键交互提示(供下游实现参考,设计侧不需要画出动态过程)**:全局目录字段与"以管理员权限运行安装"复选框在语义上联动——一旦用户填写了全局目录,管理员权限勾选框应自动置为选中,提示用户"这个选择已经默定要求管理员权限了"。设计稿只需要保证两者视觉上能各自独立展示"已勾选/未勾选"状态即可,不需要画出这次联动的动画过程。

### 4.3 提交区

- 卡片内容右下角单一主按钮:`size=lg`(48px 高,`radius-lg`),default variant(`bg-primary` + glow),左侧带一个下载类图标,文案为"开始安装 Scoop"。
- 点击后不直接开始安装,而是弹出一个**通用一次性确认弹窗**(复用设计系统 §8.4 Dialog 规格,而非专门定制的"配置摘要"弹窗):标题复述按钮文案,正文复述"这些配置对应官方脚本参数、确认后不可修改、本次配置会保存在本机"这段说明,底部 `secondary` 取消 + `default` 确认两个按钮。**这个确认弹窗不逐项复述用户刚填写的 8 项字段值**——设计上不需要额外设计一个"摘要清单"样式。

## 5. 状态 B' · 表单 + 失败重试(F16 + F19)

在状态 B 的卡片基础上,标题区下方、表单标题上方插入一条**内嵌错误提示条**(不是 Toast、不是独立 Dialog):

- 容器:`radius-md`,边框 `border-destructive/35`,背景 destructive 微光色(低透明度红),文字 `text-destructive`。
- 左侧一个告警三角图标(顶部对齐文本第一行)。
- 右侧两行文字:加粗小标题("安装失败"这一类措辞)+ 一行具体原因说明(区分"未完成需重试" / "用户已取消" / "脚本跑完但复检仍不可用"三种文案,但视觉样式三者共用同一条错误提示条,不用三种不同颜色区分)。
- 表单字段应**带入用户上一次确认过的配置值**(而不是清空重来),向用户传达"你不需要重新填一遍"。

此提示条常驻展示(不自动消失),因为它描述的是"上一次尝试"这个持续有效的事实,用户需要带着这个信息去修改配置或直接重试,不适合用几秒后自动消失的 Toast 表达。

## 6. 状态 C · 协助安装中(F16 + F17)

- 卡片头部与状态 B 共用同一套布局与配色(**图标容器仍是 warning 色调**,只是把三角告警图标换成下载类图标),标题与说明文字切换为"正在安装 Scoop…/正在执行官方安装脚本"这一类进行时文案。
- 表单区整体替换为:
  1. **不确定态进度条**(设计系统 §8.5 的简化版:`bg-secondary` 容器 + `bg-primary` 填充,填充带微光,但本页没有百分比数值来源,因此进度条必须是"扫动"动画而非可计算宽度的确定进度)。
  2. **实时日志区**:固定高度(约 260px),`font-mono`、`text-sm`、深色半透明底、边框、圆角、内部自动滚动到底部——直接对应设计系统 §8.5"原始日志区"规格。日志内容是"类终端输出"的堆叠文本(下载/初始化/写入 shim 等阶段性输出行),不分段、不加图标,纯文本堆叠最贴近真实体验。
- 底部只保留一个按钮:**取消**,`cancel` variant(中性灰,不是 destructive 红)——协助安装被取消不是失败,是用户主动终止,视觉上不能用红色。安装尘埃落定(终态)前该按钮应始终可点击。
- 本状态下**不出现**表单字段、不出现"开始安装"按钮——用户在安装执行期间不能修改本次已确认的配置。

## 7. 与设计系统通用规则的一处已知偏离(供后续统一裁决)

设计系统 §9 硬约束表规定"错误展示:错误用 Sonner Toast 或 Dialog;**不**用 inline alert"。但本页状态 B' 的失败提示条(§5)在语义上就是一个持续展示、需要与表单并存的 inline alert——一次性 Toast 会在用户还在阅读/修改表单时就消失,Dialog 又会阻断用户查看/编辑下方表单。本设计提示词按"这是一个需要长期驻留、与表单共存的状态说明"的产品意图,采用了 inline 提示条的形式。这条规则冲突留给设计系统的下一次修订统一裁决,本文档只如实记录该页面的既有呈现方式,不擅自改写设计系统条款。

## 8. 硬约束核对(设计系统 §9,本页适用项)

| 硬约束 | 本页落实方式 |
|---|---|
| 按钮禁用态 | 安装进行中,表单与"开始安装"按钮整体不渲染(而非渲染后禁用),只保留一个恒常可点击的"取消" |
| 取消态配色 | 取消按钮全程使用 `--fg-muted` 中性灰,不使用 destructive |
| i18n | 页面所有文案(标题/说明/8 项字段标签与帮助文本/按钮/错误提示)均须有 zh/en 两套,禁止单语言文案 |
| 网格背景 | 全页叠加 `.bg-grid` + 顶部 `.bg-glow-top`,与 P03~P06/P08/P09 保持同一套科技感语言 |
| 键盘 | 一次性确认弹窗需支持 `Enter` 确认、`Esc` 关闭(遵循 Dialog 通用行为) |
| 无障碍 | 表单每个输入都要有关联 `Label`;Checkbox 需可通过键盘切换 |
