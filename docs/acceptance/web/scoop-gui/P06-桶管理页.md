## 1. 范围与测试前置条件

- 覆盖功能:F09(已添加桶列表查看)、F10(已知桶清单查看)、F11(桶添加)、F12(桶移除),以及横切的 F17(任务进度反馈)、F19(错误处理与友好提示)在 P06 内的呈现。
- 测试环境两种均可,行为应一致:
  1. `npm run dev`(浏览器 mock 模式,`src/api.ts` 无 `__TAURI_INTERNALS__` 时自动降级为假数据,便于不依赖真实 Scoop 环境即可执行下列大部分用例);
  2. `npm run tauri dev`(真实 Scoop 环境,用于验证真实 `scoop bucket add/rm/list/known` 行为与 F19 的真实失败路径)。
- 前置条件统一记法:"P06 已加载且 `errors.buckets` 为空"表示已完成一次成功的 `refreshBuckets()`(即 `bucket_list`/`bucket_known` 均成功返回)。
- 除非用例中特别说明,所有用例均在"已添加"Tab 为默认可见 Tab 的前提下开始(进入页面 / 每次数据刷新后默认停留或回到"已添加")。

| AC-P06-66 | 输入的桶名/仓库地址被 Rust 侧白名单正则(`NAME_RE`/`REPO_RE`)拒绝 | 提交添加 | **不产生任何 InstallJob / P08 条目**;仅显示一条"任务创建失败:{错误信息}"的 toast;这是与 AC-P06-63(任务跑完后失败)不同的另一条错误路径,二者都应可被验证到 |

| 用例 | 前置条件 | 操作 | 预期结果 |

## 9. 与页面 PRD 的已知出入(供 PM/QA 复核,不视为本文档缺陷)

以下三点是 `docs/prd/pages/web/scoop-gui/P06-桶管理页.md` 第 5 节"验收要点"文字表述与 `src/pages/BucketsView.tsx` 真实实现之间的出入。按任务要求,验收判定**以代码现状为准**(即以本文档第 3~8 节为准),此处仅登记供 PM/QA 后续决定是否需要回头调整 PRD 文字或补开发需求:

1. PRD 写"已添加桶列表为空时,应显示 P10 空状态并引导用户前往已知区段添加桶",但代码现状(AC-P06-70)只显示纯文案,没有"引导前往已知区段"的 hint 或按钮。
2. PRD 写"已知桶清单读取失败时,应显示 P10 错误状态并允许用户重试",但代码现状(AC-P06-71)对"已知清单为空"分支同样只显示纯文案,没有独立的重试按钮——真正带重试按钮的错误态只存在于"`bucket_list`/`bucket_known` 任一请求失败"这一顶层合并错误态(AC-P06-72/73)。换言之,"已知读取失败"目前不会单独出现,它已被顶层错误态吸收;PRD 字面描述的"已知区段独立错误态"在代码里并不存在。
3. PRD 第 5 节把"添加失败/移除失败时应看到失败原因与重试入口"列为验收要点,代码现状通过 toast + P08 任务面板 + 该行自动回到可操作态三者组合满足该意图(AC-P06-63/64),而不是在 P06 表格行内维护一个独立的"失败"徽标或"重试"按钮。功能意图达成,但呈现形式与 PRD 字面描述("应看到…重试入口"容易被理解为页面内专门控件)有差异。
