## 1. 验收范围与测试前置说明

本文档覆盖页面 PRD 第 2 节功能清单(F15/F16/F17/F19)在 `BootView.tsx` + `store.ts`(Boot Sequence 编排)中的真实呈现,状态口径引用 `docs/prd/flows/scoop-gui.md` §2.2(Scoop 自身)/ §2.4(InstallJob)/ §2.6(Boot Sequence),不重新定义状态名。

### 1.1 如何在测试环境中稳定复现本页各状态

- **真机(Tauri)环境**:在已经装有 Scoop 的机器上,设置进程环境变量 `SCOOP_GUI_FORCE_SETUP=1` 后启动应用,`detect_scoop` 会强制返回 `available:false`,从而稳定复现"未检测到 Scoop"路径,无需借助一台真正未装 Scoop 的机器(见 `docs/architecture/api/scoop-gui.md` §3.4)。
- **浏览器 mock 环境**(`npm run dev`,无 `__TAURI_INTERNALS__`):在地址栏追加 `?setup` 查询参数(如 `http://localhost:5173/?setup`),`src/api.ts` 的 mock 层会将 `scoopAvailable` 初始置为 `false`,并在触发"协助安装"后用一组预设日志与约 2.4 秒的模拟延时演出完整的排队 → 执行 → 成功 → 复检通过流程,便于无 Rust 后端时端到端走查本页全部状态。
- 以上两个开关仅为测试与开发用途,均不对普通用户暴露为可见 UI 开关,验收时不应在生产界面中寻找它们的入口。

### 1.2 状态映射表(供用例引用)

| 验收用例中的状态描述 | `BootStage`(`types.ts`) | `Scoop 自身` 状态(flow §2.2) |
|---|---|---|
| 检测中 | `init` \| `detecting` | 检测中 |
| 未检测到 Scoop(表单) | `not-found` | 未安装 / 不可用(安装失败) |
| 协助安装中 | `installing` | 协助安装中 |

## 2. F15 · 启动时 Scoop 可用性检测

**TC-F15-01 检测中状态呈现**
- 前置条件:应用完成语言偏好确认,进入"检测 Scoop"阶段(`boot` 为 `init` 或 `detecting`)。
- 操作步骤:观察此阶段的页面呈现,不做任何操作。
- 预期结果:整页无侧边栏、无表单卡片,居中展示一个持续旋转的加载图标 + 主文案"正在检测本机 Scoop…" + 一行 `font-mono` 灰色辅助文案"正在执行 scoop --version 检查命令可用性";该阶段不出现任何可点击按钮。

**TC-F15-02 检测判定可用 → 直接进入主界面,不停留本页**
- 前置条件:本机 `scoop` 命令可正常调用(或使用未设置 `SCOOP_GUI_FORCE_SETUP` 的正常机器)。
- 操作步骤:正常启动应用,完成语言偏好确认。
- 预期结果:检测完成后不出现"未检测到 Scoop"表单卡片,直接进入 P03 主界面壳;不要求用户填写任何安装配置。

**TC-F15-03 检测判定不可用 → 呈现协助安装入口**
- 前置条件:`SCOOP_GUI_FORCE_SETUP=1` 或本机确无可用 `scoop` 命令。
- 操作步骤:正常启动应用,完成语言偏好确认。
- 预期结果:进入本页"未检测到 Scoop"卡片(见第 3 节),不进入 P03。

**TC-F15-04 检测动作本身异常(而非"判定为不可用")**
- 前置条件:`detect_scoop` 调用本身抛出异常(IPC 层失败,而非正常返回 `available:false`)。
- 操作步骤:触发该异常路径(需构造 IPC 失败;可作为探索性测试记录现象,而非常规回归用例)。
- 预期结果:页面表现与 TC-F15-03 完全一致(直接落到"未检测到 Scoop"表单),**不**额外展示错误提示条——真实代码在这条 `catch` 分支中只切换 `boot` 状态,不设置 `setupError`,因此这类异常与"正常判定为不可用"在页面上是不可区分的两种成因。

## 3. F16 · 安装配置表单(未检测到 Scoop)

**TC-F16-01 卡片头部呈现**
- 前置条件:`boot = not-found`,`setupError = ""`(从未尝试过安装,或本次是首次进入)。
- 预期结果:卡片左上角为警示三角图标,容器为 warning(琥珀)色调(边框 `warning/30` + 背景 `warning` 15% 透明),**不是** primary 绿色或 destructive 红色;标题"未检测到 Scoop";说明文字提示本机未安装 Scoop、将由本产品协助完成首次安装、需要网络连接。此状态下不出现错误提示条。

**TC-F16-02 八项配置字段齐全且顺序固定**
- 预期结果:表单从上到下依次为且仅为以下 8 项,不多不少:
  1. 安装根目录(ScoopDir)
  2. 全局软件包目录(ScoopGlobalDir)—— 标签旁带一枚"需要管理员权限" Badge
  3. 下载缓存目录(ScoopCacheDir)
  4. 代理服务器(Proxy)
  5. 代理例外(NoProxy)
  6. 代理用户名(ProxyCredential)与代理密码 —— 横向两列并排,仅用户名字段带说明文字,密码字段(`type="password"`)不带
  7. 使用 Windows 默认凭据访问代理(ProxyUseDefaultCredentials)—— 单行 Checkbox
  8. 以管理员权限运行安装(RunAsAdmin)—— Checkbox,下方带一行说明其行为与不勾选时的推荐做法
- 每个字段留空时应显示对应的官方默认值占位符(如 `%USERPROFILE%\scoop`),但**不会**把默认值预填为实际输入值——占位符消失的唯一方式是用户自己输入内容。

**TC-F16-03 首次进入时表单为空**
- 前置条件:本机从未通过本产品确认过安装配置(`settings.installConfig` 为空)。
- 预期结果:全部 8 项均为空 / 未勾选,仅展示占位符提示。

**TC-F16-04 曾确认过配置的场景下,表单自动带入上次的值**
- 前置条件:本机 `settings.installConfig` 已有值(此前成功或失败过至少一次协助安装确认)。
- 操作步骤:重新进入"未检测到 Scoop"表单(例如上次安装失败后)。
- 预期结果:8 项字段(含两个 Checkbox)按上次确认的值回填,用户不需要重新填写全部内容。

**TC-F16-05 填写全局软件包目录会自动勾选管理员权限**
- 前置条件:表单当前"以管理员权限运行安装"未勾选。
- 操作步骤:在"全局软件包目录(ScoopGlobalDir)"输入任意非空路径。
- 预期结果:"以管理员权限运行安装"复选框被自动置为选中;该联动只在文本非空时触发一次置选,用户此后仍可手动取消勾选(该字段不会被禁用或锁定)。

**TC-F16-06(标注实现现状,非规范应有行为)确认安装前不做字段级校验**
- 前置条件:表单任意状态。
- 操作步骤:在"安装根目录"等路径类字段中输入明显不合法的内容(例如空字符串以外的非法字符、或在"代理用户名"填写而"代理密码"留空造成的凭据不完整、或代理与 NoProxy 语义冲突的取值),然后点击"开始安装 Scoop"。
- 预期结果(按真实实现):**不会**出现任何前置校验提示,一次性确认弹窗照常弹出,确认后照常进入"协助安装中"并把这些值原样提交给 `install_scoop`。任何格式问题只会在安装脚本实际执行后,以"安装失败"这一通用结局呈现(见 TC-F16-11),而不会在提交前被单独指出具体是哪个字段、哪种冲突导致的问题。
- 说明:页面 PRD 第 5 节验收要点原文要求"用户填写无效路径、冲突代理设置或不完整代理凭据时,应在确认安装前看到可理解的校验提示",但 `BootView.tsx` 的 `submit()` 与"开始安装 Scoop"按钮均未实现任何前置校验逻辑(表单 → 直接 trim → 一次性确认弹窗 → 提交)。本用例按真实代码行为记录为"预期结果",不代表该 PRD 要求已被放弃;如需追加校验,应作为需求变更另行评审,不在本轮验收范围内视为缺陷。

**TC-F16-07 一次性确认弹窗**
- 操作步骤:填写(或不填写)表单后点击"开始安装 Scoop"。
- 预期结果:弹出居中确认弹窗,标题复述"开始安装 Scoop",正文为一段通用说明(与表单顶部的说明文字相同,强调"参数对应官方脚本 / 确认后不可修改 / 本次配置会保存在本机"),底部为"取消"(secondary)与"确认"(primary)两个按钮。**弹窗不会**逐项列出用户刚填写的 8 项字段值作为摘要。
- 点击"取消":弹窗关闭,表单保持刚才填写的内容不变,不发起任何安装。
- 点击"确认"(或聚焦弹窗后按 `Enter`):弹窗关闭,进入"协助安装中"状态(见第 5 节)。

**TC-F16-08 确认后配置本机持久化**
- 前置条件:完成一次"确认安装"。
- 预期结果:本次提交的 8 项配置被写入本机持久化设置(`installConfig`),供下次进入本表单时回填(对应 TC-F16-04)。

## 4. F16 · 表单 + 失败重试提示(未检测到 Scoop,曾尝试失败)

**TC-F16-09 失败提示条的呈现位置与样式**
- 前置条件:`boot = not-found` 且 `setupError` 为 `"failed"` / `"cancelled"` / `"recheckFailed"` 三者之一(即此前发起过至少一次协助安装,但未达成"可用")。
- 预期结果:在卡片头部说明文字下方、表单标题("安装配置")上方,出现一条**常驻的行内提示条**(不是 Sonner Toast,也不是独立弹窗):destructive 配色(红色系边框 + 低透明度红色背景 + 红色文字),左侧告警三角图标,右侧两行文字。

**TC-F16-10 三种失败成因共用同一条提示条,仅正文不同**
- 预期结果:无论 `setupError` 是 `"failed"`(安装未完成)、`"cancelled"`(用户主动取消)还是 `"recheckFailed"`(脚本跑完但复检未发现可用命令),提示条的**标题文字固定都是"安装失败"**,只有第二行的具体说明文字随成因切换为对应文案。即便是用户自己取消导致的状态,标题也不会显示为"已取消"或其它措辞——这是真实实现的现状,验收时不应因标题与"取消"语义不完全贴合而判定为缺陷。

**TC-F16-11 失败后表单带入上次配置,允许直接修改后重新提交**
- 预期结果:错误提示条下方的表单已按 TC-F16-04 回填上次确认过的配置值;用户可以直接修改任意字段后,点击**同一个**"开始安装 Scoop"按钮重新发起安装——页面**没有**另外提供一个独立标注为"重试"的按钮,"重试入口"与"修改配置入口"是同一控件(表单本身 + 该主按钮)。

**TC-F16-12 允许无限次重试**
- 预期结果:失败 / 取消 / 复检失败后,用户可以反复发起新一轮协助安装,不存在次数上限提示或锁定。

## 5. F16 + F17 · 协助安装中(页面内嵌进度)

**TC-F17-01 进入协助安装中的呈现切换**
- 前置条件:已在确认弹窗中点击"确认"。
- 预期结果:卡片头部图标切换为下载类图标,但**图标容器配色仍保持 warning(琥珀)色调**,不切换为 primary 绿色;标题变为"正在安装 Scoop…",说明文字变为"正在执行官方安装脚本,请保持网络连接。以下为实时输出:"。原表单区域整体被替换,不与进度 / 日志共存。

**TC-F17-02 不确定态进度条**
- 预期结果:呈现一条动态进度条,视觉上为"扫动"效果而非可计算宽度的百分比进度——协助安装没有可量化的完成度来源,不应呈现具体百分比数字。

**TC-F17-03 实时日志区**
- 预期结果:固定高度的深色半透明日志区,等宽字体(`font-mono`),逐行追加安装脚本的输出;新日志到达时自动滚动到底部,用户之前向上翻看的位置不影响后续追加。

**TC-F17-04 安装期间不可修改本次配置**
- 预期结果:协助安装中不出现任何表单字段,用户无法在本次任务执行期间修改已提交的安装配置;若想使用不同配置,必须等待本次任务落定终态后才能在表单里重新编辑。

**TC-F17-05 取消按钮的可用性与配色**
- 预期结果:唯一的操作入口是"取消"按钮,**中性灰色**(既不是 primary 绿色也不是 destructive 红色),对应设计系统"取消 ≠ 失败"的语义;该按钮在任务处于"排队中"或"执行中"时可点击,一旦任务进入终态(成功/失败/取消)则立即不可再点击。

**TC-F17-06 协助安装进度不会打开右下角任务浮层(P08)**
- 前置条件:任意一次协助安装进行中。
- 操作步骤:观察屏幕右下角是否出现任务浮层(`JobsPanel`,即 P08 的浮层呈现)。
- 预期结果:**不会**出现。真实代码中只有 F05/F06/F07/F11/F12(`enqueue()`)会把 `jobsPanelOpen` 置为 `true`;协助安装(`startInstallScoop()`)从不设置该状态,协助安装任务的进度只呈现在本页卡片内部,与 P08 浮层是两套互不关联的展示。此用例用于防止未来的实现把两者意外耦合或意外解耦而未被察觉。

**TC-F17-07 取消协助安装**
- 前置条件:协助安装处于"排队中"或"执行中"。
- 操作步骤:点击"取消"。
- 预期结果:任务进入"已取消";页面回到"未检测到 Scoop"表单,`setupError` 为 `"cancelled"`,错误提示条第二行文案为"安装已取消。"(标题仍是"安装失败",见 TC-F16-10)。

## 6. F16 · 安装成功与复检

**TC-F16-13 安装成功且复检通过 → 进入主界面**
- 前置条件:协助安装任务进入"已成功"。
- 操作步骤:等待页面自动发起复检(无需用户操作)。
- 预期结果:复检判定 `scoop` 可用后,页面自动离开 P01,进入 P03 主界面壳,并触发一次主界面初始数据加载(已装列表 / 过期状态 / 桶列表)。用户全程不需要手动点击"进入主界面"之类的按钮。

**TC-F16-14 安装脚本成功但复检未通过**
- 前置条件:协助安装任务进入"已成功",但复检 `scoop` 命令时仍判定不可用(或复检调用本身异常)。
- 预期结果:**不会**进入 P03;页面回到"未检测到 Scoop"表单,`setupError` 为 `"recheckFailed"`,错误提示条第二行文案为"安装脚本执行完毕,但复检未发现可用的 scoop 命令。"。

**TC-F16-15 安装任务失败**
- 前置条件:协助安装任务进入"已失败(可重试)"。
- 预期结果:页面回到"未检测到 Scoop"表单,`setupError` 为 `"failed"`,错误提示条第二行文案为"安装未能完成。你可以检查网络与配置后重试;下方日志给出了失败原因。"。

## 7. F19 · 错误处理与呈现方式

**TC-F19-01 安装结果类错误用行内常驻提示条,不用 Toast/Dialog**
- 预期结果:第 4 节描述的失败 / 取消 / 复检失败提示,统一使用页面内常驻的行内提示条呈现,**不会**以 Sonner Toast(自动消失)或独立 Dialog(遮挡表单)的形式出现。
- 说明:设计系统 `docs/design/systems/web.md` §9 硬约束表原文规定"错误展示:错误用 Sonner Toast 或 Dialog;**不**用 inline alert"。本页对"上一次协助安装的结局"这一持续性状态说明,采用的是与该条硬约束字面冲突的 inline 提示条实现。本文档按真实代码记录该行为为验收预期,该规则冲突留待设计系统下一次修订统一裁决,不在本轮验收中作为缺陷处理。

**TC-F19-02 安装任务创建本身失败时用 Toast,而非行内提示条**
- 前置条件:点击一次性确认弹窗的"确认"后,`install_scoop` 这次 IPC 调用本身即失败(尚未产生 InstallJob,例如参数序列化异常等场景)。
- 预期结果:页面**不会**进入"协助安装中",而是停留在表单态;错误以 Sonner Toast 呈现,文案模板为"任务创建失败:{具体错误}"。这与第 4 节"上一次安装已完成但结果是失败"的行内常驻提示条是两种不同的错误呈现路径,分别对应"任务从未真正开始"与"任务跑完但没达成目标"两种不同语义。

**TC-F19-03 语言持久化失败不影响本页流程**
- 说明:与 P01 无直接业务关联,仅作为交叉验证——语言切换/持久化失败的提示（Toast）不应出现在 P01 的加载路径中干扰安装表单展示。若在 P01 阶段观察到无关的语言持久化提示,应记为缺陷。

## 8. 设计系统硬约束核对(`docs/design/systems/web.md` §9,本页适用项)

| 硬约束 | 验收要点 |
|---|---|
| 按钮禁用态 | 安装进行中,除"取消"外没有其它按钮同时出现在页面上(表单与主 CTA 整体不渲染,而非渲染后置灰) |
| 取消态配色 | "取消"按钮任何时候都不得使用 destructive(红)配色 |
| i18n | 本页涉及的全部文案 key(标题、说明、8 个字段的标签与帮助文本、两个 Checkbox 文案、按钮文案、三种失败提示文案、检测态文案)在 `src/i18n.ts` 中均须同时存在 `zh` 与 `en` 两个值,不得出现只有一种语言的文案 |
| 网格背景 | 全页背景叠加 `.bg-grid`,顶部区域叠加 `.bg-glow-top`,与 P03~P06/P08/P09 视觉语言一致 |
| 键盘 | 一次性确认弹窗:`Enter` 确认、`Esc` 关闭且效果等同点击"取消" |
