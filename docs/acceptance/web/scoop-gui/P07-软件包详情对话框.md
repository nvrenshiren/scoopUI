# P07 · 验收标准 · 软件包详情对话框

> 文档层级:页面级契约(qa 产出)
> 适用范围:web 端 scoop-gui 模块 P07
> 上游依赖:`docs/prd/pages/web/scoop-gui/P07-软件包详情对话框.md`、`docs/design/prototypes/web/scoop-gui/P07-软件包详情对话框.html`
> 真相源:`src/components/DetailDialog.tsx`(如验收标准与 PRD 描述有出入,以组件真实实现为准)

## 1. 范围与数据口径

- 覆盖对象:P04(已装软件包列表页)、P05(浏览与搜索软件包页)中点击整行触发的详情对话框(全局状态 `detailName` 非空时挂载,任意时刻至多一个实例)。
- **不覆盖桶详情**:P06(桶管理页)当前实现未提供任何"桶 → 详情对话框"的交互入口,`detailName` 从未被桶管理页设置。模块 PRD 第 3 节"P07 展示单个软件包或单个桶的详细信息"的表述与页面 PRD、`flows/scoop-gui.md`、组件实现三者不一致,本文档按后三者(均不含桶详情)界定验收范围,已知偏差详见第 8 节。
- 详情数据来自单次 `scoop_info(name)` 调用(`docs/architecture/api/scoop-gui.md` §3.8),返回字段名/字段值二元组数组;**字段集合随软件包而异,不是固定 schema**,验收时不应假定任何字段必然出现。
- 业务身份由三个独立信号推导,验收时应分别构造对应前置条件:
  - `isInstalled`:该包名是否出现在已装列表(`scoop list`)中;
  - `isOutdated`:该包名是否出现在过期集合中(即 `scoop status` 判定存在比本机更新的版本);
  - `busy`:该包名当前是否有状态为"排队中"或"执行中"的任务(任一写操作种类命中即算,不区分安装/卸载/更新)。

## 2. 用例组 A —— 打开入口与头部展示

| 编号 | 前置条件 | 操作 | 预期结果 |
|---|---|---|---|
| A1 | P04 已装列表非空 | 点击任意一行的非操作按钮区域 | 打开详情对话框,展示该行对应的包名 |
| A2 | P05 可装列表非空 | 点击任意一行的非"安装"按钮区域 | 打开详情对话框,展示该行对应的包名 |
| A3 | 对话框已打开 | 观察头部 | 依次展示图标框、包名(等宽字体)、版本号(等宽字体,取自字段数组中 `Version` 键的值)、状态 Badge |
| A4 | 该包已装且过期 | 观察头部 Badge | 展示警告态 Badge"过期" |
| A5 | 该包已装且非过期 | 观察头部 Badge | 展示成功态 Badge"已安装" |
| A6 | 该包可装未装 | 观察头部 Badge | 不展示任何 Badge |

## 3. 用例组 B —— 字段展示区

| 编号 | 场景 | 预期结果 |
|---|---|---|
| B1 | 详情正在读取 | 字段区展示居中的加载指示(旋转图标 + "正在读取详情…");头部版本号此时为空 |

| B4 | 字段名未命中双语字典(scoop 对个别包返回的非常规字段名) | 标签**原样展示英文字段名**,不报错、不留空、不隐藏该行 |
| B5 | 字段为 `Description` 或 `Notes` | 值使用正文字体展示(非等宽),保留原始换行 |
| B6 | 字段为除 `Description`/`Notes` 外的其他字段 | 值使用等宽字体展示 |
| B7 | 字段为 `Website` 且值非空 | 值展示为可点击超链接,新标签页打开,悬浮下划线 |

## 5. 用例组 D —— 操作触发与流转

| 编号 | 操作 | 预期结果 |
|---|---|---|
| D1 | 点击"安装" | **不出现二次确认**,立即以安装类型入队该包名对应的任务,对话框随即关闭,任务面板(P08 承载)可见新任务 |
| D2 | 点击"更新" | **不出现二次确认**,立即以更新类型入队,对话框随即关闭 |
| D3 | 点击"卸载" | 对话框立即关闭;应用视图切换为"已装软件包列表"(P04);P04 弹出卸载二次确认对话框,标题与正文中包含目标包名;**此时尚未产生任何任务** |
| D4 | 在 D3 触发的二次确认中点击"确认卸载" | 以卸载类型入队该包名对应的任务,确认对话框关闭 |
| D5 | 在 D3 触发的二次确认中点击"取消" | 确认对话框关闭,停留在 P04,不产生任何任务 |
| D6 | 从 P05(浏览页)对一个"可装但其实已装"的行(展示"已安装" Badge 的行)打开详情并点击"卸载" | 同 D3:即便入口是 P05,也会先关闭详情对话框并把应用视图切换到 P04,再弹出二次确认——用户会被带离 P05,这是预期行为而非缺陷 |

## 8. 与上游 PRD 的已知偏差(供追溯,不代表拒绝验收现状)

本文档以 `src/components/DetailDialog.tsx` 真实实现为验收依据;以下三点与上游契约的字面表述存在出入,已按实现现状写入前述用例,特此登记供后续同步上游文档参考,**本文档不因这些偏差而判定实现不合格**:

1. **详情读取失败的"重试"入口**:页面 PRD 第 5 节验收要点原文为"详情读取失败时,用户应看到可理解的错误提示,并可选择重试或关闭",但实现只提供"关闭"这一条路径(见 B9/B10),没有独立的"重试"按钮;双语字典中 `common.retry` 词条已定义但当前未被本组件引用。**验收按现状只要求"关闭"路径可用,不要求存在专门的重试控件**。
2. **P07 不包含桶详情**:模块 PRD 第 3 节页面清单把 P07 描述为"展示单个软件包或单个桶的详细信息与可用操作",但页面级 PRD 全部 5 个交互场景、`flows/scoop-gui.md`、`BucketsView.tsx`、`DetailDialog.tsx` 均不涉及桶详情。**验收范围不包含桶详情**(见第 1 节)。
3. **三种写操作的确认不对称**:页面 PRD 第 3.3 节"出口"把安装/卸载/更新三种操作描述为对称的"触发后进入 P08",但实现中只有安装与更新是直接触发,卸载会先关闭详情并让 P04 弹出二次确认(见用例组 D)。**验收按实现现状要求这一不对称行为**,这是对破坏性操作的刻意保护,不应在测试中被当作缺陷上报。

## 9. 边界与异常补充(用例组 G)

| 编号 | 场景 | 预期结果 |
|---|---|---|
| G1 | `scoop_info` 返回空数组(理论边界,如包信息极度精简) | 字段区不报错,展示空的字段列表区域;头部版本号为空;底部按钮仍按 `isInstalled`/`isOutdated`/`busy` 正常展示 |
| G2 | 界面语言切换为英文后再打开详情 | 头部/字段标签/按钮文案均切换为英文;`scoop_info` 原始返回值(字段值本身、未收录字段名)不受语言切换影响,原样展示 |
| G3 | 同一软件包被快速连续点击打开(如双击同一行) | 只挂载一个对话框实例,不重复发起请求造成竞态覆盖异常(实现通过过期请求丢弃机制防止旧响应覆盖新状态) |
