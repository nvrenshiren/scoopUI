## 1. 前置条件

- 本机已完成首次启动流程(`flow` §1.1),Boot Sequence 处于"进入主界面",即 `Scoop 自身 = 可用` 且 `UI Language ≠ 未选择`。
- 从 P03 主界面壳侧边栏"设置"导航项进入 P09(`view: "settings"`)。
- 验收环境可读写 `%APPDATA%\scoop-gui\config.json`(本机持久化物理位置,见 `src-tauri/src/settings.rs`)。

## 2. 用例组 A · 界面语言切换(F14)

| 用例 ID | 用例描述 | 验证方法 | 期望结果 |
| --- | --- | --- | --- |
| A1 | 从 P03 设置入口可进入 P09 | 点击侧边栏"设置"导航项 | 主区切换为 P09,侧边栏"设置"项呈选中态,壳层(侧边栏/其他导航)保持可见 |
| A2 | 语言选项只有两个 | 检视语言分组下的可选控件数量与文案 | 有且仅有"中文""English"两个按钮;无第三语言、无导入语言包、无运行时翻译入口 |
| A3 | 当前语言可见 | 观察语言分组内按钮选中态 | 与本机当前 `Settings.language` 一致的按钮呈选中态(实心 + 勾选图标),另一按钮为描边态 |
| A4 | 切换语言后当前页面文案立即变化 | 当前语言为 English 时点击"中文"(或反向) | 无需任何确认/等待,P09 页面全部文案(标题、四个 Section 标题与说明、只读字段标签中"应用版本/Scoop 版本"等经 i18n 的文案)立即切换为目标语言 |
| A5 | 切换语言后全局文案同步变化 | 切换语言后,不离开 P09 情况下检视侧边栏导航文案;或切换后跳转 P04/P05/P06 | 侧边栏导航项("已装软件""浏览软件包""桶管理""设置""任务")与其余已打开视图文案同步切换,无需刷新或重启 |
| A6 | 无需重启即可继续使用新语言 | 切换语言后继续在应用内完成任意只读浏览操作 | 全程使用新语言,不出现新旧语言混排,不需要重启进程 |
| A7 | 切换后持久化,下次启动生效 | 切换语言 → 完全退出并重新启动应用 | Boot Sequence 直接以新语言进入主界面,不再呈现 P02 语言选择页(`flow` 2.5.3-3/4、2.6.3-2) |
| A8 | 未切换直接离开语言不变 | 不点击任何语言按钮,直接从 P09 导航离开 | 语言保持切换前的值,不发生任何隐式变更 |
| A9 | 持久化失败时的提示与降级 | 令 `set_language` 持久化失败(例如以只读权限锁定 `%APPDATA%\scoop-gui\config.json` 或断开该文件写入权限后触发切换) | 页面文案仍按用户本次选择立即呈现(不回滚);出现 **info 级**提示,文案为"设置已生效,但保存到本机失败;下次启动可能恢复旧值"(`settings.persistFailed`);不进入任何失败态弹窗,不阻塞后续操作 |
| A10 | 语言切换不影响业务状态 | 切换语言前后比对已装软件包列表、桶列表、任务面板中任意进行中/终态任务的状态 | 应用包 App / 桶 Bucket / InstallJob 状态机均不发生任何流转,与语言切换完全解耦 |
| A11 | 重复选择当前语言应为 no-op | 当前语言为中文时再次点击"中文" | 不触发任何持久化调用、不产生任何提示、界面无可观察变化(对应 `switchLanguage` 内 `if (lang === get().lang) return;` 守卫) |
| A12 | 语言按钮标签自身不翻译 | 分别在中文/英文界面下观察语言分组两个按钮上的文字 | 无论当前界面语言为何,两个按钮文字始终固定显示为"中文"与"English",不随界面语言切换而改变（自称名惯例） |

## 7. 与页面 PRD 的口径差异登记(据实登记,供后续评审参考,不改写 PRD 原文)

1. **主题验收用例(用例组 B)不在页面 PRD 字面范围内。** `docs/prd/pages/web/scoop-gui/P09-设置与语言切换页.md` 的功能清单只列 F14 + F19,验收要点也只覆盖语言切换。但设计系统 `docs/design/systems/web.md` §9 硬约束明确要求"P09 提供亮/暗/跟随系统三选一",且 `SettingsView.tsx` 确实完整实现了该能力。本文按"真实代码 + 设计系统硬约束"两者均已存在的事实纳入验收范围,而不是仅按 PRD 字面缩小验收面。
2. **只读回显区(用例组 C、D)不在页面 PRD 字面范围内。** 页面 PRD 全文未提及"上次确认的安装配置"或"关于"。这两块是 `SettingsView.tsx` 中真实存在、且总是渲染的内容,按"以代码为准"原则纳入验收范围。
3. **持久化失败提示的级别按真实实现校正为 info,而非 destructive/错误态。** `flow` §1.8 异常分支本身已写明"该情形仅以业务语言告知用户,本流程不进入失败态";真实代码 `switchLanguage`/`switchTheme`(`src/store.ts`)也确实统一调用 `toast.info(...)`。用例 A9/B6 按此校正验收,不应按"错误/失败"类用例处理(例如不应要求出现错误码、destructive 配色或"重试"按钮)。
4. **P09 与 P03 的关系按真实壳层结构验收(用例 E2/E3)。** 页面 PRD 与设计系统均未明确排除"Sidebar 内嵌主题/语言快捷入口"的可能性(设计系统 §8.7/§10.1 甚至描述了这样一个入口),但 `src/App.tsx` 的 `Shell` 组件里并不存在该控件——语言与主题的唯一切换入口就是 P09 本页。用例 E3 按真实壳层结构验收,如果未来实现中在 Sidebar 补出一个功能重复的快捷入口,应视为新增能力而非本页缺陷,需另行评审是否与设计系统条款保持一致。
5. **未验收"即将推出"占位区。** 本产品 Tauri 迁移前的历史版本设计稿中曾包含一个"即将推出"占位区块(自动更新/备份还原/托盘设置/Scoop 配置/shims 管理五项禁用占位)。当前真实代码未实现该区块,页面 PRD 验收要点也明确写着"不应出现 Scoop 配置编辑等未来能力入口",与模块 PRD 决策 D01 一致。用例 E1 按"不应出现"验收,不将该占位区的缺失视为缺陷。

## 8. 非阻断/建议覆盖项

以下用例涉及本机文件系统故障注入或跨进程重启,建议在具备受控测试环境时覆盖,不作为本页面上线的阻断条件:

- 持久化文件写入中途中断(断电/强杀进程模拟)后,`config.json` 不应损坏到无法被 `serde_json::from_str` 解析(`settings::save` 采用 tmp 文件 + `rename` 的原子写入,理论上可将该风险降到最低,但建议至少做一轮人工破坏性验证)。
- 首次启动即目标 `%APPDATA%\scoop-gui\` 目录不存在时,`set_language`/`set_theme` 应能自动创建目录并成功持久化(对应 `settings::save` 内 `fs::create_dir_all`)。
