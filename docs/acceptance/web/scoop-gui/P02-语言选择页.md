## 1. 范围与前置说明

- 本文档覆盖页面 PRD `P02-语言选择页.md` 第 5 节全部验收要点,以及 F13(界面语言选择·首次启动,`modules/scoop-gui.md` F13)与 F19(错误处理,涉及本页的持久化失败分支)在本页的验收表达。
- 两处"以代码为准"的澄清,贯穿全篇验收用例:
  1. **选择即确认**:页面 PRD §3.2/§4 把交互写成"用户选择其中一种 → 用户确认"两个逻辑步骤;但 `LanguagePick.tsx` 的真实实现里,点击任意一张语言卡片会直接调用 `chooseLanguage()`,没有独立的"确认"按钮或二次弹窗,也不存在"已选中但未确认"的中间态。以下用例按"点击卡片 = 选择且确认同时完成"编写。
  2. **文案来源 = 组件字面量,不是 i18n 字典**:`src/i18n.ts` 中的 `lang.title`/`lang.subtitle`/`lang.zh`/`lang.en` 四个词条在代码里没有被引用,页面真实展示的是 `LanguagePick.tsx` 硬编码的双语并列字符串。以下用例中出现的期望文案,均取自组件源码而非字典。
- 测试环境说明:
  - 真机验收优先使用打包后的应用,配置文件位于 `%APPDATA%\scoop-gui\config.json`(见项目 `CLAUDE.md`);删除该文件或清空其中 `language` 字段可复现"未持久化语言偏好"的前置状态。
  - 浏览器 mock 模式(`npm run dev`,无 `__TAURI_INTERNALS__`)下,语言持久化落在 `localStorage["mock-lang"]`;清除该 key(或使用浏览器隐私模式/清缓存)可复现同样的前置状态。mock 模式下 `set_language` 恒定成功,**无法**用来复现"保存失败"分支,该分支需在真机上人为制造写入失败(例如将配置目录设为只读)才能验证,或由 developer 阶段提供的单元/集成测试覆盖。
  - 浏览器 mock 模式下访问地址追加 `?setup` 查询参数,可让语言选择完成后稳定停在 P01"未检测到 Scoop"表单页(而不是一闪而过的检测中画面或直接进入主界面),便于稳定观察"语言切换后下一屏文案是否正确"。

## 2. 首次启动展示 P02

- 前置条件:本机(或 mock 的 `localStorage`)尚未持久化界面语言偏好。
- 操作步骤:启动应用(或刷新浏览器 mock 页面)。
- 预期结果:
  - 应用启动后 Boot Sequence 直接进入"检测语言偏好"并呈现 P02(`flow` §1.1 步骤 3、§2.5.3-1/2),而不是先出现 P01 的"检测中"画面。
  - 页面只展示"中文""English"两个选项,不出现第三个语言选项、"自动检测系统语言"入口或语言包导入入口(`project.md` §3.3、`flow` §2.5.4、PRD 验收要点)。
  - 页面标题区文案为中英并列同屏("选择界面语言 · Choose your language"/"之后可在设置中随时切换 · You can change it anytime in Settings"),不是只显示单一语言。

## 3. 语言二选一与即时生效(F13 核心)

- 前置条件:P02 已展示,本机未持久化语言偏好。
- 用例 3.1 选择中文:
  - 操作步骤:点击"中文"卡片(鼠标点击,或 Tab 聚焦后按 Enter/Space)。
  - 预期结果:
    - 点击后立即(无需二次确认)进入下一阶段——Boot Sequence 从"检测语言偏好"进入"检测 Scoop"(`flow` §2.5.3-1、§1.1 步骤 3-4)。
    - 从这一刻起,产品所有面向用户的文案立即使用中文(例如紧随其后的检测中提示"正在检测本机 Scoop…",或若走到 P01 表单页,标题应为"未检测到 Scoop"等中文文案)。
    - 不出现"已选中但未生效"的中间画面或额外的"确认"按钮。
- 用例 3.2 选择 English:
  - 操作步骤:点击"English"卡片。
  - 预期结果:
    - 同 3.1,但后续所有文案立即切换为英文(例如"Detecting Scoop on this machine…",或 P01 表单页标题"Scoop Not Detected"等)。
- 用例 3.3 未做选择不可继续:
  - 操作步骤:仅将光标悬停在卡片上、或用 Tab 聚焦但不按 Enter/Space/不点击。
  - 预期结果:Boot Sequence 保持停留在"检测语言偏好"/P02,不会自动或提前进入"检测 Scoop"阶段。

## 4. 持久化与"下次启动不再展示 P02"

- 前置条件:已完成一次语言选择(用例 3.1 或 3.2 之一),且本机持久化写入成功。
- 操作步骤:完全退出应用后重新启动(或刷新浏览器 mock 页面,不清除 `localStorage`)。
- 预期结果:
  - 不再展示 P02;Boot Sequence 直接依据已持久化的语言值进入"检测 Scoop"阶段(`flow` §1.1 步骤 2、§2.5.5)。
  - 应用启动后的文案语言与上次选择一致。

## 5. 持久化保存失败(F19)

- 前置条件:用户在 P02 完成一次语言选择,但本机持久化写入本身失败(真机场景:配置目录不可写;测试方法见第 1 节)。
- 操作步骤:在持久化会失败的前置条件下,点击任一语言卡片。
- 预期结果:
  - 语言选择依然立即生效(界面立刻切换到所选语言),Boot Sequence 依然继续推进到"检测 Scoop",**不**卡在 P02、也**不**新增第三种语言状态或错误页面(PRD 场景 4、`flow` §1.8 异常分支)。
  - 用户会看到一条业务化提示("设置已生效,但保存到本机失败;下次启动可能恢复旧值" / "Applied, but saving failed — the old value may return on next launch"),该提示可能出现在 P02 之后的下一屏(检测中/P01/P03)之上,因为提示为跨页 Toast,而语言选择后页面会立即向下一阶段推进,不会停留等待持久化结果。
  - 下一次重新启动时,由于持久化失败,语言偏好可能回退为未持久化状态,即可能重新看到 P02(而不是稳定记住本次选择)——这是该异常分支的预期表现,不是缺陷。

## 6. 边界态与硬约束覆盖

- 用例 6.1 双语文案完整性:检查 P02 页面上出现的每一处文案(标题、副标题、提示语、两张卡片的主文案与预览行),确认都同时具备中文与英文表达,不出现"半中半英"的单个句子被拆开、或某一语言缺失的情况(PRD 验收要点、design-system §9 i18n 硬约束)。
- 用例 6.2 键盘可达性:仅使用 Tab 键在两张语言卡片间移动焦点,确认每张卡片都能被聚焦并有可见聚焦样式,且聚焦后按 Enter 或 Space 能触发与鼠标点击等价的选择结果(design-system §9 键盘硬约束)。
- 用例 6.3 暗色默认 / 亮色兼容:全新首次启动(无任何持久化设置)时,P02 应以暗色主题呈现;若本机已存在持久化的 `theme=light`(但语言仍未持久化,理论边界态,可通过 mock 手动设置 `localStorage["mock-theme"]="light"` 并清空 `mock-lang` 复现),P02 应以亮色 token 正确呈现,不应出现暗色/亮色 token 混用或不可读的对比度(`src/store.ts` 的 `boot()` 在判断语言前已先应用 `theme`,因此该边界态在真实代码里是可达的)。
- 用例 6.4 无法绕过:确认不存在任何"跳过""暂不选择""稍后设置"之类的入口——语言必须二选一才能继续(`roles.md` C13、PRD 验收要点)。
- 用例 6.5 重复测试的正确复现方式:若需要反复复现"首次启动"前置态做回归测试,应完全退出已运行的 scoop-gui 进程后再启动(而不是尝试二次启动触发单实例聚焦),否则新进程不会创建、也不会重新走 Boot Sequence(`flow` §1.9 单实例规则,横切说明,不构成本页专属分支)。
