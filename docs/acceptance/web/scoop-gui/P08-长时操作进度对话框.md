## 1. 适用范围与验收依据说明

本文档验收的对象是 `JobsPanel.tsx` 组件承载的**任务进度悬浮面板**,覆盖 F05(单包安装)、F06(单包卸载)、F07(单包更新)、F08(批量更新过期)、F11(桶添加)、F12(桶移除)六类长时操作在产品运行期间的进度呈现、终态展示与取消交互。

页面 PRD 第 3.1 节把 P01(F16 协助安装 Scoop)也列为本页面入口之一,但代码层面 `src/store.ts` 的 `startInstallScoop()` 只切换 `boot` 状态、从不设置 `jobsPanelOpen`,F16 的进度/日志/取消 UI 由 `src/pages/BootView.tsx` 自行渲染成另一张居中卡片,并不经过 `JobsPanel` 组件。**本文档的验收用例不覆盖 F16/P01 场景**,该场景的验收应归入 P01 页面自身的验收文档;本文档第 10 节对此另有专项条款登记该出入,不在此处重复验收。

## 2. 入口与展示时机

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 2.1 | 在 P04/P05/P06/P07 任一处发起 F05/F06/F07/F11/F12 动作后,任务进度面板应自动展开显示 | 触发 `enqueue(kind, target, repo?)` 后,`jobsPanelOpen` 应变为 `true`,面板可见 |
| 2.2 | 从 P04 发起批量更新(F08)确认清单后,面板应展开,且清单中每个软件包各自对应一条独立任务,而不是一条"批量任务" | 清单含 N 个过期软件包时,面板任务列表应新增 N 条 `kind:"update"` 的独立任务行,而非 1 条汇总行 |
| 2.3 | 面板首次出现前(尚无任何任务)不应误报错误或空白闪烁 | 应用刚进入主界面且未发起过任何长时动作时,面板处于关闭态,侧边栏"任务"导航项不带徽标计数 |
| 2.4 | 面板挂载与 Boot Sequence 阶段无关 | 面板组件在 `App.tsx` 中始终挂载(与 `boot` 阶段无关),仅 `jobsPanelOpen` 决定是否可见;但实际只有进入主界面后由 P04/P05/P06/P07 触发的六类动作会把它打开 |

## 3. 任务列表展示

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 3.1 | 面板中应能同时展示多条任务,覆盖"排队中/执行中/已成功/已失败(可重试)/已取消"五种状态 | 人为制造 5 种状态各一条(或利用真实操作序列产生),逐条在列表中可见且徽标文案与状态一一对应(见 3.3) |
| 3.2 | 任务列表排序为按创建 id 降序(最新发起的任务排最上面),与实际执行顺序(单 worker 先进先出)无关 | 先发起 A 后发起 B,B 应显示在 A 上方,即使 A 仍在排队、B 尚未开始执行 |
| 3.3 | 状态 → 图标 → 徽标 variant → 徽标文案的映射需与源码一致,不得自行调整配色语义 | 排队中→中性灰徽标;执行中→蓝色(info)徽标 + 旋转图标;已成功→绿色徽标 + 勾选图标;已失败(可重试)→红色徽标 + 警示图标;已取消→**中性灰**徽标(与排队中同一 variant,不是单独配色)+ X 图标 |
| 3.4 | 任务标题格式固定为"<动作标签> <目标名>",两者间一个空格,不带"scoop"前缀、不带分隔符 | 例如"安装 git"“更新 nodejs”“添加桶 extras”,而非"scoop install git"或"安装 · git" |
| 3.5 | 任务标题过长时应省略号截断,不撑破面板宽度(面板固定 480px) | 用超长包名(如 40+ 字符)发起安装,标题不换行、不溢出,以省略号结尾 |
| 3.6 | 任务集合为空时,不展示列表与详情区,只展示居中空状态文案 | 所有任务清空后(理论场景,或用于设计走查),面板显示"暂无任务"/"No tasks yet",不显示表头或空表格 |

## 4. 详情 / 日志区

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 4.1 | 用户未手动点击任何任务行时,详情区应自动展示"当前执行中"的任务 | 面板刚展开、存在一条执行中任务时,详情区默认就是它,无需用户操作 |
| 4.2 | 执行中任务变为终态后,若用户此前从未手动选择过,详情区应自动切换到"新的执行中任务"(单 worker 顺序执行,下一条排队任务转正) | 任务 A(执行中)结束、任务 B(原排队中)转为执行中后,未手动点击过的前提下,详情区应从 A 自动切到 B |
| 4.3 | 用户手动点击某一行后,详情区"钉住"该行,不再随其它任务的状态变化而自动跳转 | 手动点击一条已成功/已失败的历史任务后,即使此时有另一条任务从排队转为执行中,详情区仍停留在用户选中的那条 |
| 4.4 | 不确定态进度条仅在当前详情对象为"排队中"或"执行中"时渲染,其余三种终态不渲染进度条元素(而非渲染为满格或灰色) | 切换详情对象到已成功/已失败/已取消的任务时,进度条整块消失;切回执行中任务时重新出现 |
| 4.5 | 进度条不展示数值百分比,只做不确定态扫动动画 | 全程观察不到"NN%"字样;这是因为 Scoop CLI 不产出结构化进度,不应在实现中臆造百分比 |
| 4.6 | 日志区展示当前详情对象的累计逐行输出,新行到达后自动滚动到底部 | 执行中任务持续产生新日志行时,日志区始终保持在最新一行可见,无需用户手动滚动 |
| 4.7 | 日志内容为纯文本展示,不做逐行着色或语义高亮(即便某行是错误信息) | 失败任务的日志区中,错误行与其它行样式一致,不单独标红 |
| 4.8 | 空日志(如刚提升为执行中、尚未产生输出的任务)显示占位符而非空白 | 新转为执行中但日志数组为空的任务,详情区日志区展示占位符(如"…"),不是完全空白 |

## 5. 取消行为

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 5.1 | 取消按钮仅出现在"排队中"或"执行中"的任务行,其余三种终态不出现取消按钮 | 已成功/已失败(可重试)/已取消的任务行不应存在任何取消控件 |
| 5.2 | 取消按钮使用中性灰配色,不使用 destructive(红色)配色 | 取消按钮的视觉样式应与"卸载""失败"等 destructive 语义明确区分,走中性灰系(与设计系统 §9 硬约束一致) |
| 5.3 | 点击取消后,该任务进入"已取消"终态,不进入"已失败"终态 | 取消操作后徽标文案为"已取消"/"Cancelled",不出现"已失败(可重试)"字样,也不触发失败提示 |
| 5.4 | 可同时存在多个"排队中/执行中"任务,各自的取消按钮互不影响 | 同时存在 2 条以上活跃任务时,取消其中一条不影响其余任务的运行 |
| 5.5 | 取消一条"执行中"任务后,若存在"排队中"任务,应有一条按创建顺序自动转为"执行中"(单 worker 顺序执行的体现) | 取消当前执行中任务后,观察下一条排队任务的状态在短时间内从"排队中"变为"执行中" |
| 5.6 | 取消不提供"重试"入口,与"失败"的重试路径不共用同一交互 | 已取消的任务行不出现任何"重试"按钮;这与失败任务"允许重试"的业务语义是两回事(即便面板本身对两者都不提供内联重试,见第 10 节) |

## 6. 关闭与非模态行为

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 6.1 | 面板头部右上角关闭按钮任何时候均可点击,不受任务是否执行中影响,不被禁用 | 存在执行中任务时,关闭按钮仍可正常点击,点击后面板隐藏 |
| 6.2 | 关闭面板不会中断任何正在执行或排队中的任务 | 关闭面板后重新打开(或等待事件),原先执行中/排队中的任务应继续推进或已推进到新状态,而不是被清空或取消 |
| 6.3 | 面板打开期间,点击面板范围之外的任意区域应关闭面板 | 点击侧边栏、表格行、页面标题区等面板外任意位置,面板应立即关闭 |
| 6.4 | 上述"点击外部关闭"发生时,被点击的底层元素本身不应响应这次点击(该次点击被吞掉) | 面板打开时点击背景中的某个按钮/开关,只观察到面板关闭,该按钮自身的动作**不**同时触发;需要再点一次(面板已关闭状态下)该按钮才会真正生效 |
| 6.5 | 该"外部点击关闭并吞掉"的规则对 Radix Portal 渲染的对话框(`role="dialog"`)与 Sonner Toast 不生效 | 面板打开时,若同时有确认对话框(如卸载确认)或 toast 出现,点击它们既不会关闭任务面板,也不会被吞掉,可正常与之交互 |
| 6.6 | 面板没有任何背景遮罩,底层页面在面板打开期间保持原有亮度且可正常滚动/操作(直至发生 6.3 的外部点击关闭) | 面板打开时底层页面不变暗、不模糊,鼠标可正常悬停、聚焦底层控件(直到真正点击触发关闭) |

## 7. 成功 / 失败 / 取消后的来源页联动

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 7.1 | `install`/`update`/`uninstall` 任务成功后,只刷新该单个目标对应的业务数据,不触发全量列表/状态刷新 | 观察网络行为(或 mock 环境下的调用记录):成功后只调用一次针对该包名的 `scoop_list` 查询,不调用无参数全量 `scoop_list` 或 `scoop_status` |
| 7.2 | `update`/`uninstall` 任务成功后,该目标此前的"过期"标记应从来源页消失 | 更新或卸载一个此前标记为"过期"的包成功后,返回 P04 应看到该包不再带过期徽标(update)或已从已装列表移除(uninstall) |
| 7.3 | `bucket-add`/`bucket-remove` 任务成功后,应刷新已添加桶列表;若"浏览软件包"页此前已加载过全量可装列表,应一并刷新 | 添加/移除桶成功后,P06 已添加桶列表体现变化;若之前进入过 P05 且已触发过全量加载,P05 内容相应更新 |
| 7.4 | 任务成功/失败/取消均应有对应的 Toast 提示,失败提示停留时间更长 | 成功/取消 toast 约 4 秒自动消失,失败 toast 约 6 秒自动消失(与 Sonner 默认/加长时长设定一致) |
| 7.5 | 任务失败时应自动展开任务面板(即便用户此前手动关闭过) | 一条任务从执行中变为失败时,若面板处于关闭状态,应自动重新展开,方便用户查看失败详情 |
| 7.6 | 任务取消时不应触发上述"自动展开面板"行为(取消是用户主动发起的,通常面板已经是打开的) | 取消操作本身不应作为"意外情况"额外弹出或强制展开面板 |

## 8. 批量更新与多任务并发展示

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 8.1 | 批量更新确认后,清单中的每个软件包各自生成独立任务,互不干扰 | N 个过期包确认批量更新后,面板中出现 N 条独立的 `update` 任务,分别可被单独取消、单独查看日志 |
| 8.2 | 批量更新中某一项失败,不影响其余项继续排队/执行 | 人为使某一项失败(如网络中断模拟),其余项应不受影响,按原顺序继续执行至各自终态 |
| 8.3 | 批量更新过程中取消其中一项,不影响其余项 | 取消某一条更新任务后,其余排队中/执行中任务不受影响 |
| 8.4 | 多个任务的日志互不串扰,切换详情选中对象时日志内容随之切换且不残留前一个任务的内容 | 依次点击不同任务行,日志区内容应完整替换为对应任务的日志,不出现拼接或残留 |

## 9. 国际化

| # | 验收用例 | 判定方式 |
| --- | --- | --- |
| 9.1 | 面板全部文案(标题、五态徽标、取消/关闭按钮、空状态提示、日志区前缀"任务日志")均有中英双语,且随全局语言切换立即生效 | 在设置页切换语言后,任务面板中的上述文案立即变为对应语言,无需重新打开面板或重启应用 |
| 9.2 | 任务标题中的"动作标签"随语言切换翻译,目标名(包名/桶名)不翻译 | 中文下"安装 git",英文下"Install git"——"git"本身不变 |
| 9.3 | 日志区内容不参与翻译,始终是 Scoop CLI 原始输出 | 切换界面语言前后,同一任务的日志文本逐字节不变 |
| 9.4 | 中英文案长度差异不应破坏布局(如徽标把取消按钮挤出面板可视区域) | 英文最长徽标"Failed (retryable)"与中文最长徽标"已失败(可重试)"均应完整显示,不截断、不换行错位 |

## 10. 已知不适用 / 超出范围条款(与页面 PRD 存在出入,登记而非回改上游)

| # | 条款 | 说明 |
| --- | --- | --- |
| 10.1 | **F16(协助安装 Scoop)不适用本组件的验收用例** | 页面 PRD 第 3.1 节把 P01 列为 P08 入口之一,但 `startInstallScoop()` 从不设置 `jobsPanelOpen`,F16 进度由 `BootView.tsx` 自行呈现,应归入 P01 自身验收文档,不在本文档内验收 |
| 10.2 | **面板本身不提供"重试"按钮,不验收"面板内重试"这一交互** | 页面 PRD 第 5 节验收要点写"失败时应看到...重试入口",但 `JobsPanel.tsx` 的失败任务行只有徽标,没有任何重试控件;真实的重试路径是用户返回来源页(P04/P05/P06)对同一目标重新发起一次动作(产生全新 InstallJob)。本文档第 5.6 条已按真实路径验收,此处仅作出入登记,不要求 developer 额外补一个"面板内重试按钮"来满足页面 PRD 字面表述 |
| 10.3 | **网格背景装饰(`bg-grid`/`bg-glow-top`)不适用于面板自身** | 设计系统 §9 硬约束表格把 P08 列入需要该装饰的页面清单,但该装饰实际只应用在面板背后的宿主页面(P04/P05/P06/P07 所在页面组件),不应要求 `JobsPanel.tsx` 自身携带这两个类;验收时不应因面板卡片本身没有网格/辉光背景而判定不合格 |
| 10.4 | **进度条不要求展示确定性百分比** | 设计系统 §10.4 骨架图画的是"45%"这类确定值进度条,但真实进度条恒为不确定态(见 4.5);验收不应以"是否显示准确百分比"作为判定标准 |

## 11. 文档范围声明
