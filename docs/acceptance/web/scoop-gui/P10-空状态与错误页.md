# P10 · 验收标准 · 空状态与错误页

> 文档层级:页面级契约(qa 产出)
> 适用范围:web 端 scoop-gui 模块 P10
> 上游依赖:`docs/prd/pages/web/scoop-gui/P10-空状态与错误页.md`、`docs/design/prototypes/web/scoop-gui/P10-空状态与错误页.html`
> 真相源:`src/components/EmptyState.tsx`(如验收标准与 PRD 描述有出入,以组件真实实现为准;本文档逐条标注出入点,不代表源码有缺陷需要修复)

---

## 1. 验收范围与真相源

P10 不是可以独立打开的页面,不能通过"导航到 P10"来验收。验收时实际操作的是 P03(主界面壳)、P04(已装列表)、P05(浏览/搜索)、P06(桶管理)在下列条件下的内容区呈现:

- 对应列表本身为空(无数据,非错误);
- 对应列表读取失败(命令级失败 / 网络失败 / Scoop 运行期间不可用,三者在当前实现中共用同一呈现,见第 7.3 条);
- 长时任务(InstallJob)以失败告终后,来源页与任务面板的联动呈现。

承载组件是 `src/components/EmptyState.tsx`,涉及页面是 `src/pages/InstalledView.tsx`(P04)、`src/pages/BrowseView.tsx`(P05)、`src/pages/BucketsView.tsx`(P06)、`src/components/JobsPanel.tsx`(P08 结果回落)。验收前建议先用 `npm run dev`(浏览器 mock 模式)或已装桌面版打开产品,配合本文档第 3 节给出的具体触发步骤操作。

## 2. 通用验收规则(适用于全部场景)

| 编号 | 规则 |
|---|---|
| G1 | 空状态 / 错误状态必须整体替换掉原本的表格/列表内容区,不得在保留旧内容的同时叠加一条 banner(硬约束:不用 inline alert)。 |
| G2 | 错误类呈现(`error` 分支)图标方框必须是**警示琥珀色**(`border-warning/30` + `bg-[var(--warning-soft)]` + `text-warning`),**不得是红色(destructive)**——红色只保留给场景 7 的 Toast 与任务面板。 |
| G3 | 非错误类空状态图标方框必须是中性灰(`border-border` + `bg-secondary` + `text-subtle`)。 |
| G4 | 标题文字始终使用 `--fg` 主文本色,不随错误/非错误切换颜色;只有图标方框变色。 |
| G5 | 任何空状态/错误状态下的文案,切换界面语言(中⇄英)后必须整句切换,不得出现中英混排。 |
| G6 | 错误类呈现的 `hint` 只能是业务语言("检查 scoop 命令是否正常后刷新重试"这类),不得出现原始异常字符串、堆栈、退出码。 |
| G7 | 每个 `EmptyState` 实例的图标语义必须与其 `error` 布尔值一致(见第 7.2 条:`error=true` 时即使显式传了自定义图标也不应该出现——目前代码里没有"自定义图标 + error"同时出现的调用,验收时确认这个组合没有被引入)。 |

## 3. 场景验收用例

每条用例给出:前置条件 / 触发步骤 / 预期呈现(逐字段核对) / 预期可操作项 / 验证方法。文案括注"zh / en"两版,均摘自 `src/i18n.ts`,验收时应两种语言各跑一遍。

### 用例 1 · P04 已装列表为空

- **前置条件**:本机(或 mock 数据)已装软件包数量为 0,且读取未失败。
- **预期呈现**:
  - 图标:方框中性灰,`Package` 回退图标。
  - 标题:"本机还没有通过 Scoop 安装任何软件包" / "No apps installed via Scoop yet"。
  - 辅助说明:"前往"浏览软件包"查找并安装第一个软件包" / "Head to Browse to find and install your first package"。
  - 操作区:**无按钮**——只有提示文字,"浏览软件包"是纯文本、不可点击、无下划线。
  - 头部(标题"已装软件包"/描述/检查更新按钮/更新全部过期按钮)与工具条(过滤输入框 + 刷新图标按钮)保持正常渲染,不受影响。
- **验证方法**:
  - mock 模式下,在"已装软件包"页依次卸载 mock 内置的全部 5 个软件包(7zip / git / nodejs / yazi / ClashforWindow),每次卸载成功后该行消失;全部卸载完成后应看到本用例描述的呈现。
  - 真机环境:在一台从未通过 Scoop 装过任何包的机器(或先手工 `scoop uninstall` 清空)上打开产品即可复现,无需借助开发者工具。
- **已知实现细节(非缺陷,验收时不应按"缺陷"处理)**:在过滤框中输入一个不匹配任何已装包名的关键字,即使本机确实装了软件包,也会触发**同一条**空状态(文案不变,包括"还没有安装任何软件包"这句话)。这是 `rows.length === 0` 判断在过滤结果与真实空列表两种情况下走同一分支导致的,验收时如遇到"明明装了软件包却提示未安装"的现象,先确认是否处于过滤态,若是则视为已知行为,不判为缺陷。

### 用例 2 · P05 可装列表为空(无查询)

- **前置条件**:搜索框为空,且当前已添加桶范围内没有任何可装软件包(或读取到的可装列表为空数组)。
- **预期呈现**:
  - 图标:中性灰,`Package` 回退图标。
  - 标题:"没有匹配的软件包" / "No matching packages"。
  - 辅助说明:"换个关键字,或到"桶管理"添加更多桶" / "Try another keyword, or add more buckets"。
  - 操作区:**无按钮**,搜索框本身保持可编辑、未禁用。
- **验证方法**:mock 模式下无法通过纯 UI 操作复现(mock 的可装列表固定为 8 条,不会被清空);验收时改用下方用例 3(有查询无结果)验证同一渲染分支即可,两者在代码中是同一条件(`filtered.length === 0`),不需要分别验证视觉呈现。若需验证"真正无查询也为空"的取数路径,需在真机上对一个未添加任何桶(或所添加桶内没有任何 manifest)的环境验证,或临时修改本地 mock 数据。

### 用例 3 · P05 有查询但无匹配结果

- **前置条件**:搜索框输入不存在于当前可装列表的关键字。
- **预期呈现**:与用例 2 完全一致(标题、辅助说明、图标、无按钮均相同)——这不是遗漏,是真实代码的既有行为(见 design-prompt 第 6 节场景 2/3 说明)。
- **验证方法**:mock 模式下,在"浏览软件包"页搜索框输入任意不存在的字符串(如 `zzz-not-exist`),8 条 mock 数据被过滤为 0 条,应立即看到与用例 2 完全相同的呈现;清空搜索框应恢复表格。
- **验收判定**:若发现该分支与用例 2 呈现不一致(例如多了一个"清空搜索"按钮),应视为**偏离当前代码实现**,需要与 developer 确认是否为有意变更,而不是直接判定为改进。

### 用例 4a · P06 已添加桶为空

- **前置条件**:本机已添加桶数量为 0,且读取未失败。
- **预期呈现**:
  - 图标:中性灰,显式 `Layers` 图标(唯一一个不是靠回退、而是显式传入的图标)。
  - 标题:"本机还没有添加任何桶" / "No buckets added yet"。
  - 辅助说明:**无**(该分支没有 `hint`)。
  - 操作区:**无按钮**。
  - "已添加 (0)" / "已知桶清单" 两个 Tab 仍然正常渲染在上方,可自由切换。
- **验证方法**:mock 模式下依次移除 mock 内置的 `main`、`extras` 两个桶,全部移除后应看到本用例呈现。
- **已知实现细节**:BucketsView 对"已添加"/"已知桶清单"两个 Tab 的内容渲染**没有独立的加载中占位**(不同于 P04/P05 会先判断 `loading` 再判断空/错误)。首次进入本页或手动点击刷新后、数据尚未返回前,可能会**短暂闪现**本用例或用例 4b 的空状态画面,数据到达后自动替换为真实列表。这不是失败态,也不阻塞后续渲染;验收时如观察到这种一闪而过的空状态,不应判定为缺陷,但如果空状态"卡住不消失"(数据已经返回但仍显示空状态),才是真正的缺陷。

### 用例 4b · P06 已知桶清单为空

- **前置条件**:`scoop bucket known` 返回的清单为空数组,且本次 `refreshBuckets()` 整体请求成功(未进入用例 5c 的整页失败态)。
- **预期呈现**:
  - 图标:中性灰,显式 `Layers` 图标。
  - 标题:"已知桶清单读取失败或为空" / "Known bucket list is empty or failed to load"。
  - 辅助说明:**无**。
  - 操作区:**无按钮**。
- **验证方法**:mock 模式下已知桶清单固定为 10 条、无法通过 UI 清空,本用例在 mock 模式下不可复现,需要真机环境或临时修改 mock 数据验证。
- **已知实现细节(重要,验收口径)**:标题文案字面提到"读取失败",但由于 `refreshBuckets()` 把"已添加桶"与"已知桶清单"两个请求合并为一次 `Promise.all`、共用同一个 `errors.buckets`,任一请求失败都会直接进入用例 5c 的整页失败态(替换掉整个 Tabs 区域),**不会**单独落到本用例的分支。也就是说,只要能看到本用例的画面,就说明这次数据请求其实是**成功**的,只是返回了空列表——"读取失败"四个字在当前控制流下不会是触发本用例的真实原因。验收时不要求为"已知桶清单单独失败但已添加桶列表正常"这种组合场景寻找触发路径,当前实现不支持这种组合(是否需要独立的失败态,属于产品决策,不在本次验收范围内评判)。

### 用例 5a · P04 读取失败

- **前置条件**:`scoop list` 命令级失败,或本机 Scoop 在启动后失效。
- **预期呈现**:
  - 图标:警示琥珀色,`TriangleAlert` 回退图标。
  - 标题:"无法读取数据" / "Failed to read data"。
  - 辅助说明:"本机 Scoop 当前不可用,相关数据无法读取。请检查 scoop 命令是否正常后刷新重试。" / "Scoop is currently unavailable on this machine. Check that the scoop command works, then refresh."
  - 操作区:一个"刷新" `secondary` 按钮,前缀 `RefreshCw` 图标,点击后重新触发 `refreshInstalled()` + `refreshStatus()`。
- **验证方法**:
  - mock 模式下**不可**通过 UI 直接触发(`src/api.ts` 的 `mockInvoke` 对 `scoop_list` 从不抛出异常)。可选验证路径二选一:
    1. 真机环境下人为破坏 `scoop` 可达性(例如临时改名 shim 目录或断开会导致命令失败的依赖)后启动产品,进入"已装软件包"页;
    2. 本地临时修改 `src/api.ts` 的 `scoop_list` 分支使其 `throw new Error("mock failure")`(仅用于验收,验收后必须还原,不得提交)。
  - 点击"刷新"按钮应看到操作区按钮保持可见、页面重新发起请求;若已恢复正常数据源,应替换回表格。

### 用例 5b · P05 读取失败

- 呈现结构与用例 5a 相同,标题替换为"可装软件包列表读取失败" / "Failed to load package list"(`browse.loadFailed`),辅助说明同样是 `error.scoopUnavailable`。搜索框在此状态下**保留在原位、可编辑**(不随内容区一起被替换)。验证方法同用例 5a(mock 不可复现,需真机或临时代码修改)。

### 用例 5c · P06 读取失败

- 呈现结构与用例 5a 相同,标题替换为"桶列表读取失败" / "Failed to load buckets"(`buckets.loadFailed`)。此状态会**替换掉整个 Tabs 区域**(已添加 + 已知桶清单两个 Tab 一起消失,只剩这一个错误态),页头的"刷新"/"添加自定义桶"按钮仍保留在原位。验证方法同用例 5a。

### 用例 6 · 运行期间 Scoop 整体不可用

- 本用例在真实代码中**没有独立的呈现分支**,与用例 5a/5b/5c 复用完全相同的判断条件与文案(`errors.installed` / `errors.available` / `errors.buckets`,`hint` 均为 `error.scoopUnavailable`)。验收时不需要为"命令级偶发失败"和"Scoop 整体不可用"寻找两套不同画面,凡是能验证用例 5a/5b/5c 的步骤,同时也验证了本用例。
- 唯一需要单独确认的验收点(来自页面 PRD):**不应自动跳转回 P01(协助安装页)**。验证方法:在主界面触发上述任一读取失败后,确认产品仍停留在主界面对应视图,`Boot Sequence` 未被重新触发,侧边栏导航仍可正常切换到其它未受影响的页面(例如 P04 读取失败时,切换到 P05/P06 应能正常工作,不应整个产品被锁死)。

### 用例 7 · 长时任务失败(P08 结果回落到来源页)

- **前置条件**:任意一条 InstallJob(安装/卸载/更新/添加桶/移除桶)以失败告终。mock 模式下可通过临时修改 `src/api.ts` 的 `mockRunJob` 使某个 `kind` 走向 `job.state = "failed"` 分支来复现(默认 mock 里所有任务都会成功,不天然产生失败,这也应记为已知限制:mock 层当前无法从纯 UI 操作触发一次任务失败)。
- **预期呈现(三件套,缺一不可)**:
  1. 失败瞬间弹出 Sonner Toast:红色左边线,文案 `"{title} 失败,可在任务面板查看日志后重试"` / `"{title} failed — check the log in the task panel and retry"`,持续 6000ms 后自动消失。
  2. 任务面板(P08,点击侧边栏"任务"或失败时自动 `jobsPanelOpen: true`)中对应任务行:`TriangleAlert` 红色图标 + `destructive` 徽标"已失败(可重试)"/"Failed (retryable)";点击该行展开日志区,`font-mono` 字体滚动显示原始输出到最后一行。
  3. 该任务行**没有**"重试"或"放弃"按钮(取消按钮只在排队中/执行中状态显示)。
  4. 来源页面(如"已装软件包")中被操作的那一行,busy 徽标消失,恢复到失败前的可操作状态(例如"更新"按钮重新可点击)——这是重试的真正入口。
- **验证方法**:核对以上四点全部出现,尤其要确认第 3 点"面板里确实没有重试按钮"、第 4 点"来源页按钮确实恢复可点击"——如果验收时发现面板里出现了重试按钮,或来源页按钮没有恢复,应判定为与当前实现不符,需要向 developer 确认是否为有意变更。

## 4. 双语与主题验收

- 所有用例的标题/辅助说明/按钮文案,须在中文与 English 下分别验收一遍(设置页切换语言后返回对应视图核对),不得出现半句中文半句英文。
- 暗色(默认)与亮色(备选)主题下,警示琥珀色图标方框与中性灰图标方框的对比度都应清晰可辨(参考 `docs/design/systems/web.md` §2.2 暗色对比表;亮色下无需重新核对具体对比度数值,只需目视确认警示色与中性色两者可区分)。

## 5. 回归/边界检查清单

- [ ] 空状态 / 错误状态出现时,宿主页头部(标题/描述/工具栏按钮)与工具条(搜索/过滤框)仍正常渲染、按钮可点击(除非该按钮本身依赖的数据也不可用,如"更新全部过期"在无过期项时本就应处于 disabled)。
- [ ] 错误态的"刷新"按钮点击后能重新发起请求,不产生重复请求堆叠或界面卡死。
- [ ] 空状态 / 错误状态之间来回切换(例如刷新成功后又再次失败)时,图标/文案/操作区能正确随状态切换,不残留上一态的按钮或文案。
- [ ] P06 的"已添加"与"已知桶清单"两个 Tab 的空/错误呈现相互独立,切换 Tab 不应把另一个 Tab 的状态带过来。
- [ ] 场景 7 的 Toast 出现后 6 秒内自动消失,不需要用户手动关闭;任务面板本身在用户手动关闭前保持打开。
- [ ] 未出现任何"以红色 destructive 样式渲染的读取失败空状态"(必须是警示琥珀色)——这是最容易被误实现的一点,建议每次视觉走查都单独核对一遍。
- [ ] 未出现任何"inline alert 横幅"式的错误展示(即在保留原有列表/表格内容的同时,于其上方或内部插入一条独立的错误提示条)。
